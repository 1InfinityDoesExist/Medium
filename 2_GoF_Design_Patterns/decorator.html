Intent

Attach responsibilities to an object dynamically by wrapping it in an object with the same interface that delegates and adds its own contribution. Decorators stack — wrap a wrap — composing surcharges, features, or behavior at runtime without subclassing.

The example domain

Shipping cost calculation: a base shipment (a domestic parcel at 40, an international crate at 90) can take optional add-ons — insurance (+15), express handling (+25) — in any combination. Subclassing every combination (InsuredExpressCrate, …) explodes; decorators make each add-on its own wrapper, so cost and description accumulate as you wrap.

How the pattern works

  • IShipment is the Component: getDescription(), getCost(), process().
  • DomesticParcel / InternationalCrate are Concrete Components — base descriptions and costs.
  • AbstractShipmentDecorator is the Decorator base: it implements IShipment and wraps one (protected final IShipment shipment). Its getCost() = wrapped cost + getSurcharge(); getDescription() appends the surcharge label; process() delegates then reports the addition. Subclasses supply only two hooks.
  • InsuranceDecorator (+15) / ExpressHandlingDecorator (+25) are Concrete Decorators — each defines just its surcharge and label.
  • Wrapping is nestable: new ExpressHandlingDecorator(new InsuranceDecorator(crate)) — 90 → 105 → 130.

The players

  • IShipment — Component — description/cost/process contract.
  • DomesticParcel / InternationalCrate — Concrete components — base cost 40 / 90.
  • AbstractShipmentDecorator — Decorator base — same interface + wrapped delegate + surcharge hooks.
  • InsuranceDecorator / ExpressHandlingDecorator — Concrete decorators — +15 / +25.
  • ShippingDesignPattern — Client — stacks two decorators on two bases.

Code walkthrough — every line explained

public interface IShipment {
    String getDescription();
    int getCost();
    void process();
}
  • The shared contract every layer speaks — base or wrapper — making wrapping invisible to callers.

InternationalCrate.java — a Concrete Component

public class InternationalCrate implements IShipment {
    private final String description;
    private final int cost;
    public InternationalCrate(String description) { this.description = description; this.cost = 90; }
    @Override public int getCost() { return cost; }
    @Override public void process() { System.out.println("The cost of " + description + ":" + cost); }
}
  • The undecorated base: fixed cost 90, its own description, and a process() that reports itself. DomesticParcel is the same at cost 40.

AbstractShipmentDecorator.java — the Decorator base

public abstract class AbstractShipmentDecorator implements IShipment {

    protected final IShipment shipment;

    public AbstractShipmentDecorator(IShipment shipment) { this.shipment = shipment; }

    @Override
    public String getDescription() { return shipment.getDescription() + ":" + getSurchargeLabel(); }

    @Override
    public int getCost() { return shipment.getCost() + getSurcharge(); }

    @Override
    public void process() {
        shipment.process();
        System.out.println("Added " + getSurchargeLabel() + " to " + shipment.getDescription()
                + " -> cost of " + getDescription() + ":" + getCost());
    }

    public abstract int getSurcharge();
    public abstract String getSurchargeLabel();
}
  • The double role: implements IShipment and wraps one. The wrapped field is interface-typed — it may hold a base shipment or another decorator. That's stackability.
  • getCost() is the accumulation rule: whatever's underneath + my surcharge. Description accumulates the same way. process() delegates first (inner layers report first), then reports this layer's addition.
  • The two abstract hooks mean a concrete decorator is ~10 lines.

The concrete decorators

public class InsuranceDecorator extends AbstractShipmentDecorator {
    @Override public int getSurcharge() { return 15; }
    @Override public String getSurchargeLabel() { return "Insurance"; }
}
  • Insurance +15; ExpressHandlingDecorator +25. All wrapping mechanics are inherited.

ShippingDesignPattern.java — the client

IShipment crateShipment = new ExpressHandlingDecorator(new InsuranceDecorator(new InternationalCrate("Machinery Crate")));
crateShipment.process();
  • Read inside-out: crate (90) → insured (105) → express (130). The variable is plain IShipment — the client can't tell how many layers it holds. The parcel gets the same two wraps: 40 → 55 → 80.

Why the design decisions

  • Why must the decorator implement the interface it wraps? Substitutability — a wrapped shipment goes anywhere a plain one does.
  • Why is the wrapped field interface-typed? Stackability: decorators wrap decorators, any order, any depth — every combination for free.
  • Why decorator instead of subclassing combinations? 2 add-ons → 4 subclasses; 3 → 8. Decorators are linear: one class per add-on.

Where this pattern shows up in real systems

  • Shipping/pricing surcharges (this example).
  • java.io: new BufferedInputStream(new FileInputStream(...)) — the classic stack.
  • Collections.unmodifiableList / synchronizedList; retry/metrics/logging wrappers around clients.

Same interface outside, a surcharge inside — stack as many as the shipment needs.
