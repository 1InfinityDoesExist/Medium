Intent

Define a family of interchangeable algorithms, encapsulate each in its own class, and let the context swap them at runtime. The same call produces different results depending on which strategy is plugged in.

The example domain

A navigation app scores candidate routes differently by mode: fastest penalizes distance plus traffic; scenic rewards distance and shuns traffic; eco and balanced weigh them their own ways. The user flips modes mid-session — the scoring algorithm must swap at runtime, and adding a new mode must not touch existing code.

How the pattern works

  • IRouteCostStrategy is the Strategy interface: estimateCost(distanceKm, trafficMinutes).
  • FastestRouteStrategy (+), ScenicRouteStrategy (−), EcoRouteStrategy (×), BalancedRouteStrategy (÷) — four interchangeable scoring formulas, one class each.
  • RouteContext holds the current strategy, exposes setRouteCostStrategy to swap it at runtime, and calculateRouteScore delegates.
  • The client scores with each mode in turn — same call, four different results.

The players

Code walkthrough — every line explained

public interface IRouteCostStrategy {
    int estimateCost(int distanceKm, int trafficMinutes);
}
  • The family's single contract: two route inputs → one score. Everything that varies hides behind this signature.

The concrete strategies

public class FastestRouteStrategy implements IRouteCostStrategy {
    @Override public int estimateCost(int distanceKm, int trafficMinutes) {
        return distanceKm + trafficMinutes;
    }
}
  • One formula, one class: fastest adds (total time proxy), scenic subtracts (reward distance, punish traffic), eco multiplies, balanced divides. Each is trivially simple, separately testable, and independently replaceable.

RouteContext.java — the Context

public class RouteContext {

    private IRouteCostStrategy routeCostStrategy;

    public RouteContext(IRouteCostStrategy routeCostStrategy) {
        this.routeCostStrategy = routeCostStrategy;
    }

    public void setRouteCostStrategy(IRouteCostStrategy routeCostStrategy) {
        this.routeCostStrategy = routeCostStrategy;
    }

    public int calculateRouteScore(int distanceKm, int trafficMinutes) {
        return routeCostStrategy.estimateCost(distanceKm, trafficMinutes);
    }
}
  • The field is deliberately not final — setRouteCostStrategy is the pattern's defining move: the algorithm swaps while the context lives.
  • calculateRouteScore contains zero scoring logic: pure delegation to whatever is plugged in.

The client

RouteContext routeContext = new RouteContext(new FastestRouteStrategy());
System.out.println("Fastest route (5 km, 20 min traffic) score = " + routeContext.calculateRouteScore(5, 20));   // 25

routeContext.setRouteCostStrategy(new ScenicRouteStrategy());
System.out.println("Scenic route (20 km, 5 min traffic) score = " + routeContext.calculateRouteScore(20, 5));    // 15

routeContext.setRouteCostStrategy(new EcoRouteStrategy());        // 5 * 20 = 100
routeContext.setRouteCostStrategy(new BalancedRouteStrategy());   // 20 / 5 = 4
  • One context, three swaps, four scores — the same calculateRouteScore call, four different behaviors. (The SpringApplication.run line boots the app; the pattern is the swap-and-score sequence.)

Why the design decisions

  • Why strategy objects instead of an enum + switch? Each mode's formula lives in its own class — new modes are new classes; the context never changes.
  • Why is the strategy field mutable? Runtime swapping is the point — the same context serves a user who flips modes mid-session.
  • Strategy vs State: twins structurally; in Strategy the client chooses the algorithm — in State, the states drive their own transitions.

Expected output

Fastest route (5 km, 20 min traffic) score = 25
Scenic route (20 km, 5 min traffic) score = 15
Eco route (5 km, 20 min traffic) score = 100
Balanced route (20 km, 5 min traffic) score = 4

Where this pattern shows up in real systems

  • Route-scoring modes in navigation (this example).
  • Comparator passed to Collections.sort; pricing engines; retry/backoff policies; compression codecs.
