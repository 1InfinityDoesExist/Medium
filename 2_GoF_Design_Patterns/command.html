Intent

Encapsulate a request as an object: a Command binds a receiver to one action. The invoker that runs commands knows only one method — run() — so requests can be queued, dispatched, logged, or retried without the invoker knowing what any of them do.

The example domain

A robotic-arm workcell: joints and grippers (actuators) perform steps — engage, move to a waypoint, grip a payload, disengage. Steps are queued as tasks and dispatched one at a time by the controller. The controller must not know arm-specific details; each task must carry everything it needs.

How the pattern works

  • IActuatorCommand is the Command interface — run().
  • IActuator is the Receiver contract (engage/disengage), with JointActuator and GripperActuator adding their specific motions (moveToNextWaypoint, gripPayload).
  • Four Concrete Commands — EngageActuatorCommand, DisengageActuatorCommand, MoveJointCommand, GripPayloadCommand — each bind an actuator at construction and translate run() into one receiver action.
  • TaskQueue is the Invoker: assignTask(cmd) stores a pending task, dispatchTask() runs it — knowing only run().

UML Class Diagram

Reading the diagram: the invoker holds a command typed to the interface and calls only run(). Each concrete command privately bundles its receiver; the invoker sees neither receivers nor actions.

The players

  • IActuatorCommand — Command — the one-method run() contract.
  • Engage/Disengage/MoveJoint/GripPayload commands — Concrete commands — receiver + action bound at construction.
  • IActuator, JointActuator, GripperActuator — Receivers — the hardware that does the work.
  • TaskQueue — Invoker — assign, dispatch, never peek inside.
  • CommandDesignPattern — Client — builds commands, queues, dispatches.

Code walkthrough — every line explained

public interface IActuatorCommand {
    void run();
}

public interface IActuator {
    void engage();
    void disengage();
}
  • The command contract: one zero-argument method — a command must be self-contained to be queueable. The receiver contract covers what every actuator can do; specific motions live on the concrete actuators.

JointActuator.java — a receiver

public class JointActuator implements IActuator {
    @Override public void engage() { System.out.println("Joint actuator is now engaged"); }
    @Override public void disengage() { System.out.println("Joint actuator is now disengaged"); }
    public void moveToNextWaypoint() { System.out.println("Joint moved to next waypoint"); }
}
  • The hardware stand-in: shared engage/disengage plus its joint-specific motion. GripperActuator mirrors this with gripPayload().

EngageActuatorCommand.java — a concrete command

public class EngageActuatorCommand implements IActuatorCommand {
    private IActuator actuator;
    public EngageActuatorCommand(IActuator actuator) { this.actuator = actuator; }
    @Override public void run() { actuator.engage(); }
}
  • The binding move: this actuator, this action, frozen into one object at construction. run() needs no arguments. Note it binds to the IActuator interface — engage works on any actuator — while MoveJointCommand binds to JointActuator specifically, because only joints have waypoints.

TaskQueue.java — the invoker

public class TaskQueue {
    private IActuatorCommand pendingTask;

    public void assignTask(IActuatorCommand task) { this.pendingTask = task; }

    public void dispatchTask() {
        if (pendingTask != null) {
            pendingTask.run();
        } else {
            System.out.println("No task assigned");
        }
    }
}
  • The invoker's entire worldview is run(). It cannot tell engaging from gripping — which means new command types flow through unchanged. The null guard handles an empty dispatch gracefully.

The client

IActuatorCommand engageJoint = new EngageActuatorCommand(joint);
...
taskQueue.assignTask(engageJoint);
taskQueue.dispatchTask();
  • Build the receivers, bind four commands, then assign-and-dispatch each through the queue: engage → grip → move → disengage. The workcell sequence, expressed as queued objects.

Why the design decisions

  • Why bind the receiver at construction? A zero-argument run() makes every command uniform and self-contained — exactly what queues, schedulers, and retries require.
  • Why does the invoker only know run()? Decoupling is the pattern: the controller schedules; commands carry the know-how.
  • Command vs Strategy: Strategy configures how a context does one job; Command reifies the request itself so it can travel — through queues, logs, undo stacks.

Where this pattern shows up in real systems

  • Robotic/automation task queues (this example).
  • Runnable/Callable submitted to executors — the JDK's command objects; undo stacks; message-queue payloads; Spring Batch tasklets.

A request, boxed with everything it needs — the queue just runs whatever comes next.
