Flyweight Design Pattern

ISeatClass.java — the Flyweight contract

Intent

Serve huge numbers of logical uses with few shared objects. Split state: intrinsic (shared, immutable, stored in the flyweight) vs extrinsic (per-use, passed as parameters, never stored). A pooling factory guarantees requests for the same intrinsic state return the same instance.

The example domain

An airline booking system reserves thousands of seats, but there are only a handful of seat classes — economy, business. The cabin-tier data is identical across every economy reservation; only the seat number and passenger vary per booking. One shared EconomySeatClass object serves every economy reservation; the varying details ride in as method parameters.

How the pattern works

  • ISeatClass is the Flyweight interface: reserve(seatNumber, passengerName) — the two extrinsic values are parameters.
  • EconomySeatClass / BusinessSeatClass store only the immutable intrinsic cabinTier.
  • SeatClassFactory pools flyweights in an EnumMap; getSeatClass uses computeIfAbsent — construct on first request (with a visible pool miss log), return the cached instance ever after.
  • The client makes 4 reservations and proves only 2 objects exist (== identity + pool size).

UML Class Diagram

Reading the diagram: the client asks the factory per booking; the factory pools by seat-class type (dashed arrow). Seat number and passenger flow only through reserve(...) — extrinsic state never becomes a field.

The players

  • ISeatClass — Flyweight — intrinsic tier stored; extrinsic booking details as parameters.
  • EconomySeatClass / BusinessSeatClass — Concrete flyweights — one shared instance per tier.
  • SeatClassType — The pool key (ECONOMY, BUSINESS).
  • SeatClassFactory — The pool — computeIfAbsent sharing + poolSize() for proof.
  • SeatBookingDesignPattern — Client — 4 reservations, 2 objects, identity proven.

Code walkthrough — every line explained

public interface ISeatClass {
    void reserve(String seatNumber, String passengerName);
}
  • The signature is the intrinsic/extrinsic split: seat number and passenger — the per-booking data — enter as parameters and are never stored.

EconomySeatClass.java — a concrete flyweight

public class EconomySeatClass implements ISeatClass {

    private final String cabinTier = "ECONOMY";

    @Override
    public void reserve(String seatNumber, String passengerName) {
        System.out.println("----Reserving " + cabinTier + " seat " + seatNumber + " for " + passengerName);
    }
}
  • One final intrinsic field — the tier. Immutable, hence safe to share across every economy booking, on any thread. BusinessSeatClass is identical with its own tier.

SeatClassFactory.java — the pool

public class SeatClassFactory {

    private final Map<SeatClassType, ISeatClass> pool = new EnumMap<>(SeatClassType.class);

    public ISeatClass getSeatClass(SeatClassType type) {
        return pool.computeIfAbsent(type, SeatClassFactory::create);
    }

    private static ISeatClass create(SeatClassType type) {
        System.out.println("(pool miss) creating flyweight for " + type);
        switch (type) {
        case ECONOMY: return new EconomySeatClass();
        case BUSINESS: return new BusinessSeatClass();
        default: throw new IllegalArgumentException("No flyweight registered for " + type);
        }
    }

    public int poolSize() { return pool.size(); }
}
  • computeIfAbsent is the entire sharing policy: first request for a tier constructs it (logging the miss — visible proof of when construction happens); every later request returns the cached instance. create is the only place new appears. poolSize() exists so the demo can prove the count.

SeatBookingDesignPattern.java — the client

ISeatClass booking1 = seatClassFactory.getSeatClass(SeatClassType.ECONOMY);
booking1.reserve("14A", "A. Menon");
ISeatClass booking2 = seatClassFactory.getSeatClass(SeatClassType.ECONOMY);
booking2.reserve("22F", "R. Iyer");
seatClassFactory.getSeatClass(SeatClassType.BUSINESS).reserve("2C", "S. Kapoor");
seatClassFactory.getSeatClass(SeatClassType.BUSINESS).reserve("3A", "N. Verma");

System.out.println("ECONOMY flyweight reused (booking1 == booking2): " + (booking1 == booking2));
System.out.println("Objects in pool for 4 reservations: " + seatClassFactory.poolSize());
  • Four reservations, two tiers. The output shows exactly two pool miss lines; == proves the two economy bookings share one object; poolSize() confirms: 4 reservations, 2 objects.

Why the design decisions

  • Why must cabinTier be final? Shared objects must be immutable — one caller mutating it would corrupt every other booking.
  • Why are seat number and passenger parameters? Stored, they'd make the instance stateful and unshareable. Extrinsic state flows through the flyweight, never lives in it.
  • Why an EnumMap pool keyed by an enum? Legal tiers are a closed set; the map is tiny, fast, and type-safe.

Where this pattern shows up in real systems

  • Shared seat-class/fare-class objects in booking systems (this example).
  • Integer.valueOf() — the JDK's flyweight cache; String interning; glyphs in text rendering; shared sprites in games.

Store what never changes, pass in what does — two objects serve four bookings, or four million.
