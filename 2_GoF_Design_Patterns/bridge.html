Intent

Decouple an abstraction from its implementation so the two hierarchies vary independently — connected by one composition field (the bridge) instead of welded together by inheritance. M report types × N formats costs M+N classes, not M×N.

The example domain

A reporting system exports reports in formats: summary or detailed reports, as PDF or CSV. Hard-wiring gives SummaryPdfReport, SummaryCsvReport, DetailedPdfReport, DetailedCsvReport — and every new format doubles the classes. Bridge splits the two axes: report types on one side, export formats on the other, connected by a field.

How the pattern works

  • IExportFormat is the Implementor: one method, render() — with PdfExportFormat and CsvExportFormat as concrete implementors.
  • AbstractReport is the Abstraction: it holds an IExportFormat (protected final — the bridge) and declares abstract export().
  • SummaryReport / DetailedReport are Refined Abstractions — each renders its own line, reaching across the bridge for the format name.
  • The client assembles all four combinations by passing formats into report constructors.

The players

  • AbstractReport — Abstraction — holds the implementor, declares export().
  • SummaryReport / DetailedReport — Refined abstractions — the report-type axis.
  • IExportFormat — Implementor — render().
  • PdfExportFormat / CsvExportFormat — Concrete implementors — the format axis.
  • ReportExportBridgeDemo — Client — all 2×2 combinations from 2+2 classes.

Code walkthrough — every line explained

public interface IExportFormat {
    String render();
}
  • The entire format-side contract. PdfExportFormat returns "PDF", CsvExportFormat returns "CSV" — in production these would do real serialization; neither knows reports exist.

AbstractReport.java — the Abstraction and the bridge

public abstract class AbstractReport {

    protected final IExportFormat outputFormat;

    protected AbstractReport(IExportFormat outputFormat) {
        this.outputFormat = outputFormat;
    }

    abstract public String export();
}
  • protected final IExportFormat outputFormat is the bridge: the abstraction has an implementor rather than being one. protected lets refined abstractions reach across; final fixes the pairing at construction.
  • export() is abstract — each report type defines its own rendering.

SummaryReport.java — a refined abstraction

public class SummaryReport extends AbstractReport {
    public SummaryReport(IExportFormat outputFormat) { super(outputFormat); }
    @Override public String export() {
        return "Summary report exported as : " + outputFormat.render();
    }
}
  • Its own line, with the format name fetched across the bridge. DetailedReport is the same shape with its own prefix. Neither names a concrete format.

ReportExportBridgeDemo.java — the client

AbstractReport[] reports = { new SummaryReport(new PdfExportFormat()), new SummaryReport(new CsvExportFormat()),
        new DetailedReport(new PdfExportFormat()), new DetailedReport(new CsvExportFormat()) };

for (AbstractReport report : reports) {
    System.out.println(report.export());
}
  • All four combinations of 2 report types × 2 formats — each assembled by constructor injection. No SummaryPdfReport class exists anywhere.

Why the design decisions

  • Why composition instead of inheritance? Inheritance welds type and format into one hierarchy → M×N subclasses. The bridge field lets each axis grow alone: adding XmlExportFormat instantly works with both existing reports.
  • Why protected final? protected gives subclasses direct access across the bridge; final means no half-reconfigured reports.
  • Bridge vs Strategy: look-alikes (both hold an interface field). Strategy swaps one algorithm behind a context; Bridge connects two hierarchies that both vary — report types and formats each have their own tree.

Where this pattern shows up in real systems

  • Report types × export formats (this example).
  • JDBC — the java.sql API bridged to vendor drivers; GUI toolkits bridging widgets to platform renderers; SLF4J bridged to logging backends.

Two hierarchies, one composition field between them — each side grows alone.
