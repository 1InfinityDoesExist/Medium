Intent

Pass a request along a chain of handlers; each handler decides on its own whether to resolve the request or escalate it. The sender calls only the first handler and never knows which tier will act.

The example domain

Content moderation: flagged posts arrive with a severity. LOW-severity reports are resolved automatically by an auto-filter; MEDIUM ones need a community moderator; HIGH ones a senior moderator. The reporting code shouldn't route — it files every report at one entry point, and the moderation chain escalates as needed.

How the pattern works

  • IModerationHandler is the Handler interface: handleReport(report) + setNextHandler(next).
  • Report carries a Severity (LOW/MEDIUM/HIGH) — the data every decision reads.
  • AutoFilterModerator resolves LOW, otherwise escalates (logging the hand-off); CommunityModerator resolves MEDIUM, otherwise escalates; SeniorModerator resolves HIGH — or declares the report unresolvable (the chain's end).
  • The client wires the chain once and feeds three severities into the same first handler; each is resolved at a different tier.

UML Class Diagram

Reading the diagram: the client knows one entry point. All three moderators implement the same handler interface, each holding a next reference (dashed arrows). The per-tier decisions written in each box make it a genuine chain, not a fixed pipeline.

The players

  • IModerationHandler — Handler interface — handleReport / setNextHandler.
  • Report / Severity — The request and the data driving every decision.
  • AutoFilterModerator — Tier 1 — resolves LOW, escalates otherwise.
  • CommunityModerator — Tier 2 — resolves MEDIUM, escalates otherwise.
  • SeniorModerator — Tier 3 — resolves HIGH; end of the chain.
  • ChainOfResponsibilityDesignPattern — Client — one entry point, three severities.

Code walkthrough — every line explained

public interface IModerationHandler {
    void handleReport(Report report);
    void setNextHandler(IModerationHandler nextHandler);
}

public class Report {
    private Severity severity;   // LOW | MEDIUM | HIGH
    ...
}
  • handleReport processes; setNextHandler links. Chains can be re-wired — a tier inserted or removed — without touching any moderator's logic. The report's severity is the only input each decision needs.

AutoFilterModerator.java — tier 1

@Override
public void handleReport(Report report) {
    if (report.getSeverity() == Severity.LOW) {
        System.out.println("Auto-filter moderator resolved the report.");
    } else if (nextHandler != null) {
        System.out.println("-----Escalating report to next moderator: " + nextHandler.getClass().getName());
        nextHandler.handleReport(report);
    }
}
  • The genuine decision: LOW → resolve here, chain over. Anything else → escalate, with a visible log naming the next tier. The null guard makes the handler safe even as the chain's tail.

CommunityModerator.java — tier 2

  • Identical shape: resolves MEDIUM, escalates everything else. Reached only when tier 1 declined.

SeniorModerator.java — tier 3

@Override
public void handleReport(Report report) {
    if (report.getSeverity() == Severity.HIGH) {
        System.out.println("Senior moderator resolved the report.");
    } else {
        System.out.println("Report could not be resolved by any moderator.");
    }
}
  • The chain's end: resolves HIGH — and handles the fell-off-the-end case explicitly rather than silently dropping a report. Its setNextHandler is a no-op; nothing follows.

The client

autoFilterModerator.setNextHandler(communityModerator);
communityModerator.setNextHandler(seniorModerator);

autoFilterModerator.handleReport(new Report(Severity.LOW));     // resolved at tier 1
autoFilterModerator.handleReport(new Report(Severity.MEDIUM));  // tier 1 -> tier 2
autoFilterModerator.handleReport(new Report(Severity.HIGH));    // tier 1 -> tier 2 -> tier 3
  • The chain is wired once. All three reports enter through the same call on the same first handler — the report's severity, not the caller, decides how far each travels.

Why the design decisions

  • Why does routing live inside the handlers? That is the pattern: the sender must not know which tier resolves a report. Three identical calls, three different journeys.
  • Why setNextHandler instead of constructor wiring? Chains get re-shaped — inserting an ML-moderator tier between auto-filter and community is one line at composition time.
  • Why does the last tier handle the unresolvable case? A request falling silently off the chain's end is the classic CoR bug; the explicit else makes the end of the line visible.

Where this pattern shows up in real systems

  • Content-moderation escalation (this example); support-ticket escalation.
  • Servlet filters and Spring Security filter chains; middleware stacks; approval workflows.

One entry point, autonomous tiers — severity decides how far a report travels.
