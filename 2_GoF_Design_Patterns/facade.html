Facade Design Pattern

The subsystems

Intent

Provide a single, simplified interface to a set of subsystems. The Facade owns and coordinates the complex parts internally; the client makes intention-revealing calls and never learns the subsystems exist.

The example domain

A smart irrigation system: watering the garden really means opening a valve, activating a soil-moisture sensor, and raising the flow rate — in the right order; stopping means closing the valve and deactivating the sensor. Gardeners shouldn't choreograph hardware. The facade turns the choreography into startWatering() / stopWatering().

How the pattern works

  • ValveController and SoilMoistureSensor are two independent subsystems with their own small operations.
  • IrrigationFacade constructs its own subsystems (private fields — unreachable from outside) and exposes two intention-revealing methods, each encoding the correct multi-step sequence.
  • The client creates the facade and makes two calls; the hardware dance is invisible.

The players

  • IrrigationFacade — The Facade — owns the subsystems, exposes startWatering/stopWatering.
  • ValveController — Subsystem 1 — open/close/increase flow.
  • SoilMoistureSensor — Subsystem 2 — activate/deactivate.
  • SmartIrrigationDesignPattern — Client — two calls, zero subsystem knowledge.

Code walkthrough — every line explained

public class ValveController {
    public void open() { System.out.println("-----Valve Controller Open"); }
    public void close() { System.out.println("-----Valve Controller Closed"); }
    public void increaseFlowRate() { System.out.println("-----Valve Controller Flow Increased"); }
}

public class SoilMoistureSensor {
    public void activate() { System.out.println("-----Soil Moisture Sensor On"); }
    public void deactivate() { System.out.println("-----Soil Moisture Sensor Off"); }
}
  • Two focused classes, each a stand-in for real hardware drivers. Neither knows the other exists; coordination is not their job.

IrrigationFacade.java

public class IrrigationFacade {

    private ValveController valveController;
    private SoilMoistureSensor soilMoistureSensor;

    public IrrigationFacade() {
        this.valveController = new ValveController();
        this.soilMoistureSensor = new SoilMoistureSensor();
    }

    public void startWatering() {
        valveController.open();
        soilMoistureSensor.activate();
        valveController.increaseFlowRate();
    }

    public void stopWatering() {
        valveController.close();
        soilMoistureSensor.deactivate();
    }
}
  • The facade builds its own subsystems as private fields — the client never sees, supplies, or touches them. That is what makes the simplification real.
  • startWatering() encodes the correct order: open the valve, activate the sensor, then raise the flow — raising flow before opening would be a hardware fault. stopWatering() encodes the shutdown sequence. The order lives in exactly one place.

SmartIrrigationDesignPattern.java — the client

IrrigationFacade irrigationFacade = new IrrigationFacade();
irrigationFacade.startWatering();
irrigationFacade.stopWatering();
  • Three lines that read as intent: make a facade, start watering, stop watering. Zero hardware knowledge.

Why the design decisions

  • Why does the facade construct its own subsystems? If the client passed them in, it would need to know they exist — the exact coupling Facade removes.
  • Why coarse methods instead of exposing the five hardware operations? The encoded order is the value: one method makes the correct sequence the only possible sequence.
  • Facade vs Adapter vs Decorator: Adapter changes an interface's shape; Decorator adds behavior behind the same interface; Facade reduces many interfaces to one.

Where this pattern shows up in real systems

  • Hardware orchestration behind one control call (this example).
  • SLF4J's LoggerFactory, Spring's JdbcTemplate — multi-object dances behind one entry point; checkout facades over inventory+payment+shipping.

Two calls in, a whole choreography behind them — and the gardener never has to know.
