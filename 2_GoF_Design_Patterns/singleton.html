Intent

Guarantee that a class has exactly one instance and provide a global, thread-safe access point to it. The class itself controls its own instantiation: no outside code can ever construct a second copy.

The example domain

An application's license key is activated once and must be identical everywhere it's checked — two different "active keys" in one process would be a correctness bug, and re-reading the license store on every check would be wasteful. LicenseKeyRegistry models this: one instance, created lazily, holding the single activated key that every caller shares.

How the pattern works

This implementation uses the Initialization-on-Demand Holder idiom (the Bill Pugh Singleton):

  • The constructor is private — only the class can construct itself.
  • A private static final nested class, InstanceHolder, holds the single instance.
  • The JVM initializes a class exactly once, on first use. Since InstanceHolder is touched only inside getInstance(), the instance is created lazily — and the JVM's class-initialization lock makes it thread-safe with zero synchronization code.
  • final on the class blocks subclassing, closing the only other route to a second instance.

The players

  • LicenseKeyRegistry — The Singleton — final class, private constructor, holder idiom, one shared key.
  • InstanceHolder — Private static nested class whose only job is holding the one static final instance.
  • SingletonDesignPattern — Client — retrieves the instance twice and proves both references are identical.

Code walkthrough — every line explained

public final class LicenseKeyRegistry {

    private final String activeLicenseKey;

    private LicenseKeyRegistry() {
        this.activeLicenseKey = "ENT-7734-9A2F-PROD";
    }

    private static final class InstanceHolder {
        private static final LicenseKeyRegistry INSTANCE = new LicenseKeyRegistry();
    }

    public static LicenseKeyRegistry getInstance() {
        return InstanceHolder.INSTANCE;
    }

    public String getActiveLicenseKey() {
        return activeLicenseKey;
    }
}
  • public final class — final blocks subclassing: a subclass could add a public constructor and mint additional instances, silently breaking the guarantee. Sealing the class closes that loophole.
  • private final String activeLicenseKey — the singleton's payload: set once in the constructor, immutable forever. In production this would be loaded from a license store; here it's a fixed key so the demo is deterministic.
  • private LicenseKeyRegistry() — the cornerstone. With the constructor hidden, new LicenseKeyRegistry() is illegal everywhere outside this class. The class alone decides when — and how many times — it is built.
  • private static final class InstanceHolder — exists purely to exploit JVM class-loading semantics: a class initializes exactly once, only when first referenced. Nothing references InstanceHolder until getInstance() runs — so the singleton is lazy for free.
  • private static final LicenseKeyRegistry INSTANCE = new LicenseKeyRegistry(); — the one and only construction in the entire program. The JVM's class-initialization lock guarantees this line runs exactly once even under heavy concurrency — no synchronized, no volatile, no double-checked locking.
  • getInstance() — the global access point. Reading InstanceHolder.INSTANCE triggers the holder's initialization on first call; every later call returns the same reference straight from the initialized class.
  • getActiveLicenseKey() — the useful work: every caller reads the same activated key from the same instance.

SingletonDesignPattern.java — the client

LicenseKeyRegistry first = LicenseKeyRegistry.getInstance();
LicenseKeyRegistry second = LicenseKeyRegistry.getInstance();

System.out.println("Both references point to the same instance: " + (first == second));
System.out.println("Active license key: " + first.getActiveLicenseKey());
  • Two retrievals; the first constructs (via the holder), the second just returns the cached reference. == compares object identity — printing true is runtime proof of the single-instance guarantee. The second line shows the shared payload.

Why the holder idiom (vs the alternatives)

  • Eager static final field on the class — No — Yes — Yes
  • synchronized getInstance() — Yes — Yes — Slow — locks every call
  • Double-checked locking + volatile — Yes — Yes (Java 5+) — Noisy, easy to get wrong
  • Initialization-on-Demand Holder — Yes — Yes — Yes — zero lock code

Where this pattern shows up in real systems

  • License/activation registries (this example); application configuration holders.
  • Runtime.getRuntime() in the JDK; logger factories; connection pools; metrics registries.
