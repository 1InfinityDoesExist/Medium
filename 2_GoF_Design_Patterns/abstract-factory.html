Abstract Factory Design Pattern

Intent

Provide an interface for creating families of related objects without naming their concrete classes. Choosing the factory chooses a whole matched set — mixing products from different families becomes impossible by construction.

The example domain

A camping-gear outfitter equips expeditions: a desert trip needs a desert tent and a desert sleeping bag; a mountain trip needs the mountain versions of both. Pairing a mountain tent with a desert sleeping bag would be an outfitting error — the tent and bag for one terrain belong together. One factory per terrain guarantees a consistent kit.

How the pattern works

  • ITent and ISleepingBag are the two abstract products — the roles every family supplies.
  • IExpeditionGearFactory is the abstract factory: createTent() + createSleepingBag(); one instance = one terrain's family. Each factory also self-describes via getExpeditionTerrain().
  • DesertGearFactory / MountainGearFactory are concrete factories — Spring @Components whose products are injected, each returning only its own terrain's gear.
  • ExpeditionGearFactoryProvider indexes all factories into an EnumMap by terrain (self-registration via Spring's list injection) and serves getFactory(terrain).
  • The client asks for the mountain family, uses it; asks for the desert family, uses it — never naming a concrete class.

Reading the diagram: the client calls getFactory(terrain) and receives one family; from it, createTent() and createSleepingBag() yield matched products. The two concrete factories each implement the family interface — one per terrain.

The players

  • ITent / ISleepingBag — Abstract products — the two roles in every kit.
  • DesertTent, DesertSleepingBag — The desert family's products.
  • MountainTent, MountainSleepingBag — The mountain family's products.
  • IExpeditionGearFactory — Abstract factory — one instance = one consistent kit.
  • DesertGearFactory / MountainGearFactory — Concrete factories — matched sets per terrain.
  • ExpeditionGearFactoryProvider — Indexes factories by terrain; getFactory(...) serves families.
  • AbstractFactoryDesignPattern — Client — uses two families through the abstractions only.

Code walkthrough — every line explained

public interface ITent {
    ExpeditionTerrain getExpeditionTerrain();
    void pitch();
}
public interface ISleepingBag {
    ExpeditionTerrain getExpeditionTerrain();
    void unroll();
}
  • Each declares one role's behavior (pitch, unroll) plus a self-describing terrain — products carry their own identity.

One concrete product

@Component
public class MountainTent implements ITent {
    @Override public void pitch() { System.out.println("Pitching a mountain tent!"); }
    @Override public ExpeditionTerrain getExpeditionTerrain() { return ExpeditionTerrain.MOUNTAIN; }
}
  • Terrain-specific behavior; the other three products are identical in shape. @Component lets Spring construct and inject them.

IExpeditionGearFactory — the abstract factory

public interface IExpeditionGearFactory {
    ExpeditionTerrain getExpeditionTerrain();
    ITent createTent();
    ISleepingBag createSleepingBag();
}
  • One accessor per product role — holding an instance means holding a whole kit. getExpeditionTerrain() lets each factory self-describe, which powers automatic registration below.

MountainGearFactory — a concrete factory

@Component
@RequiredArgsConstructor
public class MountainGearFactory implements IExpeditionGearFactory {

    private final MountainTent mountainTent;
    private final MountainSleepingBag mountainSleepingBag;

    @Override public ITent createTent() { return mountainTent; }
    @Override public ISleepingBag createSleepingBag() { return mountainSleepingBag; }
    @Override public ExpeditionTerrain getExpeditionTerrain() { return ExpeditionTerrain.MOUNTAIN; }
}
  • Spring injects the mountain products (Lombok's @RequiredArgsConstructor generates the constructor); the factory simply hands out its matched set. Only mountain gear can ever leave this factory — consistency by construction.

ExpeditionGearFactoryProvider — self-registering index

public ExpeditionGearFactoryProvider(List<? extends IExpeditionGearFactory> factories) {
    this.factoryMap = factories.stream().collect(Collectors.toMap(IExpeditionGearFactory::getExpeditionTerrain,
            Function.identity(), (a, b) -> b, () -> new EnumMap<>(ExpeditionTerrain.class)));
}

public IExpeditionGearFactory getFactory(final ExpeditionTerrain expeditionTerrain) {
    return factoryMap.get(expeditionTerrain);
}
  • Spring injects every IExpeditionGearFactory bean as a list; the constructor indexes them by their self-described terrain into an EnumMap. A new terrain's factory registers itself just by existing — no switch, no manual registry.

The client

IExpeditionGearFactory mountainGearFactory = provider.getFactory(ExpeditionTerrain.MOUNTAIN);
mountainGearFactory.createTent().pitch();
mountainGearFactory.createSleepingBag().unroll();

IExpeditionGearFactory desertGearFactory = provider.getFactory(ExpeditionTerrain.DESERT);
desertGearFactory.createTent().pitch();
desertGearFactory.createSleepingBag().unroll();
  • Two terrains, the same two calls each — and the outputs are guaranteed-matched kits. No concrete class names anywhere in the client.

Why the design decisions

  • Why Abstract Factory, not two Factory Methods? The tent and bag must match. One family object makes a mixed kit unrepresentable.
  • Why the self-describing getExpeditionTerrain()? It turns registration into data: the provider builds its index from the factories themselves; adding JUNGLE gear = new products + one new factory, nothing edited.
  • Factory Method vs Abstract Factory: choosing a Factory-Method creator picks one product; choosing an Abstract Factory picks a whole family.

Expected output

Pitching a mountain tent!
Unrolling a mountain sleeping bag!

Where this pattern shows up in real systems

  • Matched equipment kits (this example); UI theme factories (matched widgets).
  • JDBC vendors (matched connections/statements/results); cloud SDK provider factories (matched storage+queue+auth clients).

Choose the terrain, and the whole kit matches — no mixing, by construction.
