Iterator Design Pattern

The contracts

⚡ In a nutshell — Traverse an aggregate's elements sequentially without exposing its internal representation.

Intent

Traverse an aggregate's elements sequentially without exposing its internal representation. All traversal state lives in the iterator — so multiple independent walks can run at once, through hasNext() / next() only.

The example domain

A kanban task board holds tickets in priority order — P0 first. Consumers walk the board ticket by ticket without touching its internal list: the board hands out an iterator, and the client just loops.

How the pattern works

  • ITicketIterator<T> is the Iterator contract: hasNext() / next().
  • ITaskBoard<T> is the Aggregate contract: openIterator() — the factory method.
  • TaskBoard stores the tickets; TaskBoardIterator holds the cursor — the entire traversal state — and reads tickets through the board's accessors.
  • The client builds a board, opens an iterator, and drains it.

The players

  • ITicketIterator<T> — Iterator contract — hasNext/next.
  • ITaskBoard<T> — Aggregate contract — openIterator().
  • TaskBoard — Concrete aggregate — the ticket list, no traversal state.
  • TaskBoardIterator — Concrete iterator — the cursor.
  • TaskBoardDesignPattern — Client — loop and print.

Code walkthrough — every line explained

public interface ITicketIterator<T> {
    boolean hasNext();
    T next();
}

public interface ITaskBoard<T> {
    ITicketIterator<T> openIterator();
}
  • Generic in T, so boards of any ticket type share the contracts. The aggregate's single duty toward traversal: hand out iterators.

TaskBoard.java — the aggregate

public class TaskBoard implements ITaskBoard<String> {
    private final List<String> tickets = new ArrayList<>();

    public void addTicket(String ticket) { tickets.add(ticket); }
    public String getTicketAt(int index) { return tickets.get(index); }
    public int getTicketCount() { return tickets.size(); }

    @Override
    public ITicketIterator<String> openIterator() {
        return new TaskBoardIterator(this);
    }
}
  • The board stores tickets and exposes indexed accessors — but holds no cursor, no traversal state. openIterator() is the factory method: every call returns a fresh, independent iterator.

TaskBoardIterator.java — the iterator

public class TaskBoardIterator implements ITicketIterator<String> {
    private final TaskBoard board;
    private int cursor = 0;

    @Override public boolean hasNext() { return cursor < board.getTicketCount(); }
    @Override public String next() { return board.getTicketAt(cursor++); }
}
  • cursor is the entire traversal state, living here — two iterators over the same board advance independently. next() reads the current ticket and advances in one expression.

The client

TaskBoard board = new TaskBoard();
board.addTicket("P0: Fix payment gateway timeout");
board.addTicket("P1: Add dark mode toggle");
board.addTicket("P2: Refactor logging module");

ITicketIterator<String> iterator = board.openIterator();
while (iterator.hasNext()) {
    System.out.println(" 🎫 " + iterator.next());
}
  • Build the board in priority order, open an iterator, drain it — the client never touches the ticket list itself.

Why the design decisions

  • Why does the cursor live in the iterator? Independent concurrent walks, and a board that stays clean of traversal concerns.
  • Why a factory method on the aggregate? The client stays decoupled from the concrete iterator class; the board controls how it can be walked.
  • Why generic contracts? One pair of interfaces serves any element type — the same shape as java.util.Iterator/Iterable.

Where this pattern shows up in real systems

  • Kanban/queue walkers (this example).
  • Every java.util collection's iterator(); the for-each loop; database cursors; paginated API iterators.

The board stores; the iterator walks — and each walk is its own.
