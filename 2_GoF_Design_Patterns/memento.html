Intent

Capture an object's state in an opaque snapshot for later restore — without violating encapsulation. The originator creates and reads snapshots; the caretaker stores them but can never look inside — enforced here by Java package-private access, not convention.

The example domain

A shopping cart before a risky bulk update: checkpoint the cart, try the bulk add (40 clearance travel mugs), and if it was a mistake — roll back to the checkpoint. The checkpoint service stores snapshots but must never parse or modify a cart's contents; that knowledge belongs to the cart alone.

How the pattern works

  • ShoppingCart (originator) holds the manifest; createSnapshot() freezes it, restoreSnapshot() writes it back.
  • CartSnapshot (memento) has a package-private constructor and getter — only same-package code (the cart) can create or read one.
  • CartCheckpoint (caretaker) lives in the caretaker package — it stores and returns snapshots, and code that tried to peek inside would not compile.
  • The client checkpoints twice, then rolls back to the first checkpoint.

Reading the diagram: the originator creates and reads snapshots (dashed arrow); the caretaker, in another package, only stores them — the opaqueness is compiler-enforced.

The players

  • ShoppingCart — Originator — owns the manifest, snapshots and restores itself.
  • CartSnapshot — Memento — package-private access: opaque to everyone but the cart.
  • CartCheckpoint — Caretaker — stores snapshots from another package; structurally blind.
  • ShoppingCartRollbackDemo — Client — checkpoint, bulk-add, roll back.

Code walkthrough — every line explained

public class ShoppingCart {

    private String manifest;

    public ShoppingCart(String manifest) { this.manifest = manifest; }

    public void addItem(String line) { this.manifest += line; }

    public String getManifest() { return this.manifest; }

    public CartSnapshot createSnapshot() {
        return new CartSnapshot(this.manifest);
    }

    public void restoreSnapshot(CartSnapshot snapshot) {
        this.manifest = snapshot.getSavedManifest();
    }
}
  • The cart's state is its manifest — a running list of item lines. addItem appends.
  • createSnapshot() freezes the current manifest into a memento; restoreSnapshot() overwrites the manifest from one. The cart can call the snapshot's package-private members because they share the originator package — it is the only class with that privilege.

CartSnapshot.java — the Memento

public class CartSnapshot {

    private String manifest;

    CartSnapshot(String manifest) { this.manifest = manifest; }

    String getSavedManifest() { return this.manifest; }
}
  • No public on the constructor or getter — package-private. Any class outside originator that tries snapshot.getSavedManifest() fails to compile. The class itself is public so other packages can hold references — they just can't see inside. Encapsulation by compiler, not by trust.

CartCheckpoint.java — the Caretaker

package com.design.patterns.memento.caretaker;

public class CartCheckpoint {

    private List<CartSnapshot> snapshots;

    public void saveSnapshot(CartSnapshot snapshot) { this.snapshots.add(snapshot); }

    public CartSnapshot getSnapshot(int index) { return this.snapshots.get(index); }
}
  • Note the package: caretaker, not originator. It stores snapshots in a list and returns them by index — and that is all it can do. Calling the snapshot's getter here would be a compile error.

The client

ShoppingCart cart = new ShoppingCart("2x Espresso Beans 1kg\n");
cart.addItem("1x Pour-Over Filter Pack\n");
checkpoint.saveSnapshot(cart.createSnapshot());     // checkpoint #0

cart.addItem("40x Clearance Travel Mugs\n");        // the risky bulk add
checkpoint.saveSnapshot(cart.createSnapshot());     // checkpoint #1

cart.restoreSnapshot(checkpoint.getSnapshot(0));    // roll back
System.out.println(cart.getManifest());             // mugs gone
  • Two checkpoints bracket the bulk add; restoring checkpoint #0 erases the 40 mugs. The final print proves the rollback.

Why the design decisions

  • Why package-private memento members? "Please don't read the snapshot" erodes; access control makes the caretaker's blindness a compile-time guarantee.
  • Why does the caretaker live in a different package? Same-package code shares package-private access — moving CartCheckpoint out is what actually locks the door.
  • Why snapshot the whole manifest? Total restore: partial restores are the classic memento bug.

Where this pattern shows up in real systems

  • Cart/order checkpoints before bulk operations (this example).
  • Editor undo stacks; database savepoints; VM checkpoints; game save states.

Snapshot what only you may see; let others carry it, never read it.
