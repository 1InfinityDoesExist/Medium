Intent

Define a class per grammar rule, each with an interpret() operation, so a sentence in the language becomes a tree of expression objects that evaluates itself recursively. Terminals are the leaves; non-terminals combine sub-expressions; a parser turns text into the tree.

The example domain

A tiny arithmetic language: expressions like 5 + 3 - 2. The grammar has one terminal rule (a number) and two non-terminal rules (addition, subtraction). A Context parses the text left-to-right into an abstract syntax tree; one interpret() on the root computes the result.

How the pattern works

  • IExpression is the grammar contract: interpret() returns this node's integer value.
  • NumberExpression is the terminal: it interprets to its own number — the base case.
  • AddExpression / SubtractExpression are non-terminals: each holds left and right sub-expressions (typed to the interface — recursion in the type system) and interprets to left ± right.
  • Context is the parser: splits "5 + 3 - 2" into tokens and folds them into a left-deep tree — ((5 + 3) - 2) — respecting left-to-right evaluation.

The players

  • IExpression — Abstract expression — interpret() : int.
  • NumberExpression — Terminal — a literal number.
  • AddExpression / SubtractExpression — Non-terminals — combine two sub-expressions.
  • Context — Parser — text → left-deep AST.
  • InterpreterDesignPattern — Client — parse once, interpret once.

Code walkthrough — every line explained

public interface IExpression {
    int interpret();
}
  • One method returning this node's value. Every grammar rule implements it, which is what lets nodes nest arbitrarily.

NumberExpression.java — the terminal

public class NumberExpression implements IExpression {
    private int number;
    public NumberExpression(int number) { this.number = number; }
    @Override public int interpret() { return number; }
}
  • Wraps a literal; interpreting returns it. This is where recursion bottoms out.

AddExpression.java — a non-terminal

public class AddExpression implements IExpression {

    private IExpression leftExpression;
    private IExpression rightExpression;

    public AddExpression(IExpression leftExpression, IExpression rightExpression) { ... }

    @Override
    public int interpret() {
        return leftExpression.interpret() + rightExpression.interpret();
    }
}
  • Operands typed IExpression — each may be a number or another expression: the grammar's recursion in the type system. Interpreting recursively evaluates both sides and adds. SubtractExpression is identical with -.

Context.java — the parser

public IExpression parseExpression(String expression) {
    String[] tokens = expression.split(" ");
    IExpression result = new NumberExpression(Integer.parseInt(tokens[0]));
    for (int i = 1; i < tokens.length - 1; i += 2) {
        String operator = tokens[i];
        IExpression right = new NumberExpression(Integer.parseInt(tokens[i + 1]));
        if (operator.equals("+")) {
            result = new AddExpression(result, right);
        } else if (operator.equals("-")) {
            result = new SubtractExpression(result, right);
        } else {
            throw new IllegalArgumentException("Unsupported operator: " + operator);
        }
    }
    return result;
}
  • Tokenize on spaces; seed with the first number; then fold operator + number pairs into the accumulated tree. Because each new node wraps the previous result as its left child, the tree is left-deep — 5 + 3 - 2 becomes Subtract(Add(5, 3), 2), exactly left-to-right arithmetic. Unknown operators fail fast.

InterpreterDesignPattern.java — the client

String inputExpression = "5 + 3 - 2";
Context context = new Context();
IExpression expression = context.parseExpression(inputExpression);
int result = expression.interpret();
System.out.println("Result: " + result);
  • Parse once into a tree; interpret once on the root; the tree computes 6 inside-out.

Why the design decisions

  • Why a class per rule? New rules (multiply, parentheses) are new classes; existing rules stay untouched, and each rule's evaluation lives in one place.
  • Why a left-deep fold in the parser? It encodes left-to-right evaluation for same-precedence operators — 5 + 3 - 2 must be (5+3)-2, not 5+(3-2).
  • Interpreter vs Composite: structural cousins — but here each class is a grammar rule and the tree is a sentence, with a parser producing it from text.

Where this pattern shows up in real systems

  • Arithmetic/formula evaluators (this example); spreadsheet formulas.
  • Spring Expression Language (SpEL); regex engines; rule engines; SQL/query-DSL parsers.

Grammar as classes, sentence as a tree — parse once, and it evaluates itself.
