Intent

Create new objects by copying an existing instance — the prototype — instead of constructing from scratch. The copy is a genuinely independent object: mutating it never affects the original.

Prototype matters when an object is expensive to assemble (loaded from a database, computed, or configured through many steps) and you need many near-identical variants of it.

How it mirrors production code

Schema-driven platforms hold entity schemas — the definition of an entity's attributes — that are expensive to load and validate. A common operation is "make me a variant of this schema": same base attributes, plus one or two extra fields for a tenant-specific extension. Re-loading and re-validating from the source every time would be wasteful; mutating the shared original would corrupt every other consumer. The production answer is exactly this module: clone the loaded schema (deep-copying its mutable attribute list), then customize the copy freely.

How the pattern works

  • EntitySchema is the prototype: it knows how to copy itself via cloneSchema(). Crucially, the mutable attributes list is copied into a new ArrayList — a deep copy of the mutable state — while the immutable schemaId string is safely shared.
  • CloningService is a small singleton façade over cloning: callers ask it for a copy, and it delegates to the prototype's own cloneSchema(). The object being copied stays in charge of how it is copied.
  • The client builds one original, clones it, mutates only the clone, and proves the original is untouched.

Code walkthrough — every line explained

public class EntitySchema {

    private String schemaId;
    private List<String> attributes;

    public EntitySchema(String schemaId, List<String> attributes) {
        this.schemaId = schemaId;
        this.attributes = attributes;
    }

    public EntitySchema cloneSchema() {
        return new EntitySchema(schemaId, new ArrayList<>(attributes));
    }
    ...
}
  • Two fields: an identifier and a mutable list of attribute names. The list is what makes copying non-trivial — a naive copy that shared the list would let a change to the clone leak into the original.
  • cloneSchema() is the heart of the pattern. It builds a brand-new EntitySchema, passing schemaId as-is (a String is immutable, sharing the reference is safe) and — critically — new ArrayList<>(attributes): a fresh list containing the same elements. After this line, original and clone hold two different list objects. That one new ArrayList<> is the difference between a correct deep copy and a subtle shared-state bug.

CloningService.java — the cloning entry point

public class CloningService {

    private CloningService() {}

    private static final class InstanceHolder {
        private static final CloningService INSTANCE = new CloningService();
    }

    public static CloningService getInstance() {
        return InstanceHolder.INSTANCE;
    }

    public EntitySchema clone(EntitySchema source) {
        return source.cloneSchema();
    }
}
  • A tiny holder-idiom singleton (private constructor, lazy thread-safe instance) — a service like this needs exactly one shared instance.
  • clone(source) contains the pattern's key delegation: it does not reach into the schema's fields and copy them itself. It calls source.cloneSchema() — the object being copied stays the single authority on how deep its own copy must go. If EntitySchema later gains another mutable field, only cloneSchema() changes; every caller keeps working.

PrototypeDesignPattern.java — the client

EntitySchema original = new EntitySchema("schema-1",
        new ArrayList<>(Arrays.asList("name", "email")));
EntitySchema copy = CloningService.getInstance().clone(original);
copy.getAttributes().add("phone");
System.out.println("original attrs: " + original.getAttributes());
System.out.println("copy attrs: " + copy.getAttributes());
System.out.println("different instances: " + (original != copy));
  • Builds the original with two attributes, clones it, then mutates only the copy by adding "phone".
  • The three prints are the proof: the original still shows [name, email], the copy shows [name, email, phone], and original != copy confirms two distinct objects. If cloneSchema() had shared the list, the first line would (wrongly) show the phone attribute too — this demo is designed to catch exactly that bug.

Why the design decisions

  • Why new ArrayList<>(attributes) inside cloneSchema()? That is the deep-copy step for the mutable state. Immutable String can be shared; a mutable List must not be.
  • Why does the object clone itself, rather than the service copying fields? Only the class itself reliably knows which of its fields are mutable and how deep the copy must go. Centralizing that in cloneSchema() keeps the knowledge in one place.
  • Why a singleton CloningService at all? It gives the application one uniform entry point for cloning (a natural seam for later adding validation, metrics, or copy-on-write policies) without touching the prototype class.

Where this pattern shows up in real systems

  • Schema/template variants — clone a loaded entity schema, add tenant-specific fields (this example).
  • Object.clone() / copy constructors / List.copyOf idioms across the JDK.
  • Game engines cloning configured unit/enemy templates; document systems cloning report templates; HTTP request templates cloned then parameterized per call.
