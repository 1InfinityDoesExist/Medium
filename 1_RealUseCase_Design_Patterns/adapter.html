Intent

Convert the interface of an existing class into the interface a client expects. Adapter lets classes work together that couldn't otherwise, because their interfaces don't match — without changing either side.

The client depends on a clean Target interface; a legacy or engine-specific Adaptee speaks a different vocabulary; the Adapter implements the Target by wrapping the Adaptee and translating between the two.

How it mirrors production code

Declarative data platforms let users say "provision this model" without caring which engine hosts it. But each engine has its own native vocabulary — MongoDB thinks in db.createCollection(...) statements, Postgres in DDL. The planning layer wants one clean call: plan(modelId). This module models the production bridge between the two worlds: IDbSubstrateTarget is what the planner (client) wants, IDbSubstrateAdaptee is the engine-native world, and DeclarativeDbSubstrateAdapter translates engine-native statements into the declarative plan format the client expects.

How the pattern works

  • IDbSubstrateTarget — the Target: the single, clean method the client depends on: plan(modelId).
  • IDbSubstrateAdaptee — the Adaptee interface: engine-native vocabulary (substrate(), nativeStatements(name)), with MongoSubstrateAdaptee as a concrete engine.
  • DeclarativeDbSubstrateAdapter — the Adapter: implements the Target, holds the Adaptee by composition, and translates each native statement into a plan artifact.
  • The client declares only IDbSubstrateTarget; after the one composition-root line, it never touches the adaptee world.

Reading the diagram: the client depends only on the Target interface (top-left arrow). The Adapter implements the Target and holds the Adaptee by composition (dashed arrow) — an object adapter. All translation lives in one place: the adapter's plan method.

The players

  • IDbSubstrateTarget — Target — the clean plan(modelId) interface the client expects.
  • IDbSubstrateAdaptee — Adaptee contract — engine-native substrate() + nativeStatements(name).
  • MongoSubstrateAdaptee — Concrete adaptee — emits Mongo-native db.createCollection statements.
  • EngineStatement — The adaptee world's data shape (object name + native command).
  • DeclarativeDbSubstrateAdapter — Adapter — implements Target, wraps Adaptee, translates.
  • AdapterDesignPattern — Client — depends only on the Target.

Code walkthrough — every line explained

public interface IDbSubstrateTarget {
    String plan(String modelId);
}
  • One method — the whole contract the client wants. Nothing engine-specific leaks in.

IDbSubstrateAdaptee.java + EngineStatement.java — the Adaptee world

public interface IDbSubstrateAdaptee {
    String substrate();
    List<EngineStatement> nativeStatements(String name);
}
  • An incompatible vocabulary: instead of one plan string, it exposes an engine name and a list of native statements. EngineStatement is that world's data shape — an object name plus a raw native command.

MongoSubstrateAdaptee.java — a concrete Adaptee

public class MongoSubstrateAdaptee implements IDbSubstrateAdaptee {
    @Override public String substrate() { return "MONGODB"; }
    @Override public List<EngineStatement> nativeStatements(String name) {
        return Arrays.asList(new EngineStatement(name, "db.createCollection(\"" + name + "\")"));
    }
}
  • Speaks pure Mongo: creating a model means one db.createCollection("...") statement. It knows nothing about the Target interface.

DeclarativeDbSubstrateAdapter.java — the Adapter

public class DeclarativeDbSubstrateAdapter implements IDbSubstrateTarget {

    private final IDbSubstrateAdaptee adaptee;

    public DeclarativeDbSubstrateAdapter(IDbSubstrateAdaptee adaptee) {
        this.adaptee = adaptee;
    }

    @Override
    public String plan(String modelId) {
        List<String> artifacts = new ArrayList<>();
        for (EngineStatement statement : adaptee.nativeStatements(modelId)) {
            artifacts.add(adaptee.substrate() + ":" + statement.getCommand());
        }
        return "plan(" + artifacts + ")";
    }
}
  • implements IDbSubstrateTarget — from the client's point of view, the adapter is a target.
  • private final IDbSubstrateAdaptee adaptee — composition, not inheritance: the adapter has an adaptee (an object adapter). Typed to the adaptee interface, so the same adapter works for any engine.
  • plan(modelId) is the translation: ask the adaptee for its native statements, prefix each command with the engine name, and package the result in the declarative plan(...) format. Two vocabularies, one translator, in one place.

AdapterDesignPattern.java — the client

IDbSubstrateTarget target = new DeclarativeDbSubstrateAdapter(new MongoSubstrateAdaptee());
System.out.println(target.plan("customers"));
  • The composition root is the only line that names concrete classes. The variable is typed IDbSubstrateTarget — from here on, the client calls plan and has no idea Mongo exists.

Why the design decisions

  • Why an object adapter (composition) rather than a class adapter (inheritance)? Composition lets one adapter serve any IDbSubstrateAdaptee — swap in a PostgresSubstrateAdaptee and nothing else changes. Java's single inheritance makes class adapters brittle anyway.
  • Why is the adaptee field typed to the interface? So the adapter is written once for all engines — the pattern composes with polymorphism instead of multiplying adapters.
  • Why keep EngineStatement un-adapted? It never crosses the boundary: the adapter consumes it internally and exposes only the Target's vocabulary. Translation is total at the seam.

Expected output

$ java -cp target/classes com.realusecase.patterns.AdapterDesignPattern

plan([MONGODB:db.createCollection("customers")])

Where this pattern shows up in real systems

  • Declarative provisioning layers over engine-native APIs (this example).
  • java.io.InputStreamReader — adapts a byte InputStream to the Reader (character) interface.
  • Arrays.asList() — adapts an array to the List interface.
  • SLF4J bindings — adapt Log4j/JUL loggers to one facade; payment-gateway SDK wrappers adapting vendor APIs to one internal interface.

Same object, new interface — translation lives in exactly one place.
