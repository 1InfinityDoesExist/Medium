Facade Design Pattern

The subsystems

Intent

Provide a single, simplified interface to a set of subsystems. The Facade owns and coordinates the complex parts internally; the client makes one call and never learns the subsystems exist.

Facade doesn't add behavior (like Decorator) or translate interfaces (like Adapter) — it orchestrates, replacing a multi-step dance across several classes with one intention-revealing method.

How it mirrors production code

Query-orchestration services routinely run this sequence: resolve which schema version to hit, run two queries against the store, then intersect the two result-ID sets (the Redis SINTERSTORE idiom) to find records matching both filters. Forcing every caller to know those three subsystems — and their order — spreads fragile knowledge everywhere. The production fix is one facade method: executeAndIntersect(schemaId, queryA, queryB). Callers state what they want; the facade owns how.

How the pattern works

  • VersionResolver, QueryService, RedisSetMerger are three independent subsystems, each with its own small job.
  • QueryFacade is the Facade: it constructs its own subsystem instances (private fields — the client can't reach them) and its one public method runs the whole pipeline in the right order.
  • The client creates the facade and makes exactly one call.

UML Class Diagram

Reading the diagram: the client makes one simple call to the facade; the numbered arrows show the orchestration the facade performs internally — resolve, query twice, intersect. No arrow ever connects the client to a subsystem.

The players

  • QueryFacade — The Facade — owns the subsystems, exposes one intention-revealing method.
  • VersionResolver — Subsystem 1 — maps a schema id to its active version.
  • QueryService — Subsystem 2 — executes a query against a database type.
  • RedisSetMerger — Subsystem 3 — intersects two result-ID lists into a set.
  • FacadeDesignPattern — Client — one facade, one call.

Code walkthrough — every line explained

public class VersionResolver {
    public String resolve(String schemaId) { return schemaId + "-v1"; }
}

public class QueryService {
    public List<String> execute(String databaseType, String query) {
        return Arrays.asList(databaseType + "-id-1", databaseType + "-id-2");
    }
}

public class RedisSetMerger {
    public Set<String> intersect(List<String> firstResultIds, List<String> secondResultIds) {
        Set<String> result = new LinkedHashSet<>(firstResultIds);
        result.retainAll(secondResultIds);
        return result;
    }
}
  • Three focused classes, each a stand-in for a real component: a schema-registry lookup, a Mongo query executor, and a Redis set-intersection. Note RedisSetMerger uses LinkedHashSet + retainAll — a genuine intersection preserving first-list order.
  • None of them knows the others exist. Coordination is not their job.

QueryFacade.java — the Facade

public class QueryFacade {

    private final QueryService queryService = new QueryService();
    private final RedisSetMerger redisSetMerger = new RedisSetMerger();
    private final VersionResolver versionResolver = new VersionResolver();

    public Set<String> executeAndIntersect(String schemaId, String queryA, String queryB) {
        String version = versionResolver.resolve(schemaId);
        System.out.println("resolved version: " + version);
        List<String> resultsA = queryService.execute("mongo", queryA);
        List<String> resultsB = queryService.execute("mongo", queryB);
        return redisSetMerger.intersect(resultsA, resultsB);
    }
}
  • The facade constructs its own subsystems as private final fields — the client never sees, supplies, or touches them. This is what makes the simplification real rather than cosmetic.
  • executeAndIntersect encodes the entire workflow in one place: resolve the version, run both queries, intersect the results. If the pipeline someday needs caching or a fourth step, exactly one method changes — every caller is untouched.

FacadeDesignPattern.java — the client

QueryFacade facade = new QueryFacade();
System.out.println(facade.executeAndIntersect("schema-1", "q1", "q2"));
  • Two lines. The client's code reads as its intent — "execute and intersect" — with zero knowledge of versions, query services, or set merging.

Why the design decisions

  • Why does the facade build its own subsystems? If the client passed them in, it would need to know they exist — the exact coupling Facade removes. (In a Spring service the same idea appears as injected private dependencies; the point is they're not the caller's concern.)
  • Why one coarse method instead of three fine ones? The value is the encoded order. Three public steps would let callers skip or reorder them; one method makes the correct sequence the only possible sequence.
  • Facade vs Adapter vs Decorator: Adapter changes an interface's shape; Decorator adds behavior behind the same interface; Facade reduces many interfaces to one. Here nothing is translated or wrapped — three subsystems are orchestrated.

Where this pattern shows up in real systems

  • Query orchestration across resolver + store + merger (this example).
  • javax.faces.context.FacesContext, SLF4J's LoggerFactory, Spring's JdbcTemplate — each hides a multi-object dance behind one entry point.
  • Checkout services orchestrating inventory + payment + shipping; trip-booking facades over flight/hotel/car subsystems.

One call in, a whole pipeline behind it — and the client never has to know.
