Intent

Define the skeleton of an algorithm in a base-class method — with the step order fixed and non-overridable — while letting subclasses customize only specific hook steps. Shared logic is written once; variation is confined to exactly the steps meant to vary.

How it mirrors production code

Data-ingestion pipelines process record payloads for different destinations. Every destination needs the same normalization first — lower-casing all keys — but each destination then needs its own extra shaping: BigQuery, for example, requires column-safe names (first name → first_name), while an in-memory context store takes keys as-is. The production shape is a base processor whose processPayload runs normalize-then-transform in a fixed order, with transform as the per-destination hook.

How the pattern works

  • AbstractPayloadProcessor.processPayload(...) is the template method — declared final, so the algorithm's order (normalize → transform) can never be changed by a subclass.
  • Step 1 (shared): lower-case every key of every row — written once, inherited by all.
  • Step 2 (hook): transform(...) — protected, with a sensible default (identity), so subclasses override it only if they need to.
  • ContextPayloadProcessor inherits the default hook; BigQueryPayloadProcessor overrides it to sanitize keys into [a-z0-9_].
  • The client calls the same processPayload on both processors and gets destination-appropriate output.

Reading the diagram: the final skeleton lives in the abstract class; the hook is protected with a default. One subclass inherits the hook, the other overrides it — neither can touch the skeleton.

  • ContextPayloadProcessor — Concrete subclass — inherits the identity hook untouched.
  • BigQueryPayloadProcessor — Concrete subclass — overrides the hook to sanitize keys.
  • TemplateMethodDesignPattern — Client — same call, two destinations, two shapes.

Code walkthrough — every line explained

public abstract class AbstractPayloadProcessor {

    public final List<Map<String, Object>> processPayload(List<Map<String, Object>> data) {
        List<Map<String, Object>> lowered = new ArrayList<>();
        for (Map<String, Object> row : data) {
            Map<String, Object> out = new LinkedHashMap<>();
            for (Map.Entry<String, Object> entry : row.entrySet()) {
                out.put(entry.getKey().toLowerCase(), entry.getValue());
            }
            lowered.add(out);
        }
        return transform(lowered);
    }

    protected List<Map<String, Object>> transform(List<Map<String, Object>> data) {
        return data;
    }

    public abstract String resourceType();
}
  • public final on processPayload is the pattern's signature move: the skeleton — normalize first, then transform — is law. No subclass can reorder or skip the normalization.
  • The shared step copies each row into a LinkedHashMap with lower-cased keys (preserving column order, never mutating the input).
  • transform is the hook: protected (an extension point for subclasses, not an API for callers) with an identity default — a subclass with no extra shaping needs zero code.
  • resourceType() is abstract metadata: each processor names its destination.

ContextPayloadProcessor.java — inheriting the default

public class ContextPayloadProcessor extends AbstractPayloadProcessor {
    @Override public String resourceType() { return "CONTEXT"; }
}
  • The minimal subclass: names itself and changes nothing else. It still gets the full normalize-then-transform pipeline — with the identity hook.

BigQueryPayloadProcessor.java — overriding the hook

@Override
protected List<Map<String, Object>> transform(List<Map<String, Object>> data) {
    List<Map<String, Object>> sanitized = new ArrayList<>();
    for (Map<String, Object> row : data) {
        Map<String, Object> out = new LinkedHashMap<>();
        for (Map.Entry<String, Object> entry : row.entrySet()) {
            out.put(entry.getKey().replaceAll("[^a-z0-9_]", "_"), entry.getValue());
        }
        sanitized.add(out);
    }
    return sanitized;
}
  • The one varying step: every character that isn't a lower-case letter, digit, or underscore becomes _ — first name → first_name, a BigQuery-legal column. Note it can assume keys are already lower-cased: the final skeleton guarantees normalization ran first.

The client

row.put("First Name", "avinash");

System.out.println(context.resourceType() + " -> " + context.processPayload(data));
System.out.println(bigquery.resourceType() + " -> " + bigquery.processPayload(data));
  • Deliberately messy input — mixed case, a space in a key. The same processPayload call yields first name (lower-cased only) for CONTEXT and first_name (lower-cased and sanitized) for BIGQUERY.

Why the design decisions

  • Why final on the template method? The fixed order is the contract. If a subclass could override the skeleton, "normalize always happens first" would be a hope, not a guarantee.
  • Why give the hook a default instead of making it abstract? Optional variation: destinations that need nothing extra (CONTEXT) write nothing. Only genuinely varying steps burden subclasses.
  • Why can BigQuery's regex assume lower-case? Because the skeleton guarantees the normalize step already ran — hooks can rely on the steps before them. That's the power of a fixed order.
