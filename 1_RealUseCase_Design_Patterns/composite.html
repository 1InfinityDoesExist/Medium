Intent

Compose objects into tree structures representing part-whole hierarchies, and let clients treat individual objects and compositions of objects uniformly — one shared interface, one recursive operation, arbitrary nesting.

The client never asks "is this a leaf or a container?" — it just calls the operation, and the tree takes care of itself.

How it mirrors production code

Filter builders in UIs let users compose conditions visually: age > 18 AND (country = IN OR country = US). A single condition and a bracketed group of conditions must both render to SQL, and groups can nest inside groups without limit. This module models exactly that: UICondition is one atomic comparison; UIComboCondition is a bracketed AND/OR group holding any mix of atomic conditions and nested groups; and toSql() renders the whole tree with correct parentheses — the same structure query-builder libraries and rule engines use in production.

How the pattern works

  • AbstractUICondition is the Component: the one shared contract, toSql().
  • UICondition is the Leaf: an atomic field operator 'value' comparison — no children.
  • UIComboCondition is the Composite: it holds List<AbstractUICondition> — typed to the component, so children may be leaves or other composites — and implements toSql() by recursively delegating to every child and joining with its AND/OR operator inside parentheses.
  • The client builds a nested tree and calls one method on the root; recursion does the rest.

UML Class Diagram

Reading the diagram: both the leaf and the composite extend the same component. The composite's child list is typed to the component — the dashed self-referencing arrow — which is exactly what makes unlimited nesting possible: a group can contain groups.

The players

  • AbstractUICondition — Component — the shared toSql() contract.
  • UICondition — Leaf — one atomic field op 'value' comparison.
  • UIComboCondition — Composite — an AND/OR group of child components, recursively rendered.
  • Op — The combinator enum (AND, OR).
  • CompositeDesignPattern — Client — builds a nested tree, calls toSql() once on the root.

Code walkthrough — every line explained

public abstract class AbstractUICondition {
    public abstract String toSql();
}
  • The entire shared contract in one method. Everything in the tree — leaf or group — is an AbstractUICondition that can render itself to SQL. This is the uniformity the pattern promises.

UICondition.java — the Leaf

public class UICondition extends AbstractUICondition {

    private final String field;
    private final String operator;
    private final String value;

    public UICondition(String field, String operator, String value) { ... }

    @Override
    public String toSql() {
        return field + " " + operator + " '" + value + "'";
    }
}
  • Three immutable parts — field, operator, value — and a toSql() that renders them as one atomic comparison like age > '18'. No children, no recursion: this is where the tree bottoms out.

UIComboCondition.java — the Composite

public class UIComboCondition extends AbstractUICondition {

    private final Op comboOp;
    private final List<AbstractUICondition> conditions;

    public UIComboCondition(Op comboOp, AbstractUICondition... conditions) {
        this.comboOp = comboOp;
        this.conditions = new ArrayList<>(Arrays.asList(conditions));
    }

    @Override
    public String toSql() {
        return "(" + conditions.stream().map(AbstractUICondition::toSql)
                .collect(Collectors.joining(" " + comboOp.name() + " ")) + ")";
    }
}
  • The child list is typed List<AbstractUICondition> — the component type, not a concrete type. That single decision is the whole pattern: a child can be a UICondition or another UIComboCondition, so groups nest without limit.
  • The varargs constructor makes tree-building read naturally at the call site.
  • toSql() is the recursive heart: map every child to its toSql() (leaves render themselves; nested groups recurse again), join the results with AND / OR, and wrap in parentheses so nested groups keep correct precedence.

CompositeDesignPattern.java — the client

AbstractUICondition tree = new UIComboCondition(Op.AND,
        new UICondition("age", ">", "18"),
        new UIComboCondition(Op.OR,
                new UICondition("country", "=", "IN"),
                new UICondition("country", "=", "US")));
System.out.println(tree.toSql());
  • The construction visually is the tree: an AND group containing one leaf and a nested OR group of two leaves. The root is declared as the abstract component type, and one toSql() call renders everything — the client never distinguishes leaf from group.

Why the design decisions

  • Why type children as the component? So leaves and groups are interchangeable inside any group — that is what makes nesting unlimited and the client code uniform.
  • Why parentheses in the composite's toSql()? Correct boolean precedence when groups nest: (age > '18' AND (country = 'IN' OR country = 'US')) means exactly what the tree shape says.
  • Why an enum for the combinator? AND/OR are the only legal group operators; an enum makes illegal combinators unrepresentable and renders itself via name().

Where this pattern shows up in real systems

  • Visual filter/query builders rendering to SQL or Mongo filters (this example).
  • File systems (files/directories), UI component trees, org charts.
  • JSON/XML document models; Elasticsearch bool-query builders nesting must/should clauses.

One interface, one recursive call — the tree renders itself, however deep it goes.
