Response = Status Code + Header + Body

The notes open the Exception Handling chapter with this foundation: whatever a controller returns, the actual HTTP Response always has three parts —

  • Status Code
  • Header
  • Body

ResponseEntity<T> is Spring's class to create the response with full control over all three:

return ResponseEntity.status(HttpStatus.OK).headers(headers).body("Hi");

Body should be last. It is actually kind of using the Builder design pattern: status(...) and headers(...) all return a builder object, and the body(...) method call returns the actual ResponseEntity object.

// the builder's terminal methods:
<T> ResponseEntity<T> body(T body);       // sets the body and builds
<T> ResponseEntity<T> build();            // builds with NO body

For a response without a body, use ResponseEntity<Void> and finish with build():

return ResponseEntity.status(HttpStatus.OK).build();

By default, 200 is the status code set when you do not specify one.

@RestController
@RequestMapping("/api/users")
public class UserController {

    // 1) full form — status + headers + body (body MUST be last)
    @GetMapping("/{id}")
    public ResponseEntity<String> getUser(@PathVariable String id) {
        HttpHeaders headers = new HttpHeaders();
        headers.add("X-Request-Handled-By", "user-service");
        return ResponseEntity.status(HttpStatus.OK).headers(headers).body("Hi");
    }

    // 2) shortcut — 200 OK with a body
    @GetMapping
    public ResponseEntity<List<String>> allUsers() {
        return ResponseEntity.ok(List.of("alex", "maria"));
    }

    // 3) 201 Created + Location header, after creating a resource
    @PostMapping
    public ResponseEntity<String> createUser(@RequestBody String user) {
        return ResponseEntity.status(HttpStatus.CREATED)
                .header("Location", "/api/users/123")
                .body("created");
    }

    // 4) no body at all — ResponseEntity<Void> + build()
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> deleteUser(@PathVariable String id) {
        return ResponseEntity.status(HttpStatus.NO_CONTENT).build();
    }
}

Added: besides ok(...), the builder has a static shortcut for every common status — know these, they read much better than status(HttpStatus.X):

ResponseEntity.ok(body);                  // 200
ResponseEntity.created(location);         // 201 + sets the Location header for you
ResponseEntity.accepted().build();        // 202
ResponseEntity.noContent().build();       // 204
ResponseEntity.badRequest().body(err);    // 400
ResponseEntity.notFound().build();        // 404
ResponseEntity.unprocessableEntity().body(err); // 422

The canonical POST → 201 pattern builds the Location URI from the current request instead of hard-coding it:

@PostMapping
public ResponseEntity<User> createUser(@RequestBody User user) {
    User saved = userService.save(user);
    URI location = ServletUriComponentsBuilder.fromCurrentRequest()
            .path("/{id}")                       // /api/users + /123
            .buildAndExpand(saved.getId())
            .toUri();
    return ResponseEntity.created(location).body(saved);   // 201 + Location: /api/users/123
}

@ResponseBody

  • When we return a plain string or POJO directly from the class, then the @ResponseBody annotation is required.
  • @RestController automatically puts @ResponseBody on all the methods.
  • If you use plain @Controller, it will throw an Exception if the return type is a String or other object — it treats the response as a view (a page name to render).
@Controller
public class PageController {

    @GetMapping("/name")
    @ResponseBody                 // without this, "Alex" is looked up as a VIEW name
    public String name() {
        return "Alex";
    }
}

@RestController                   // = @Controller + @ResponseBody on every method
public class UserApi {

    @GetMapping("/user")
    public User user() {
        return new User("Alex");  // serialized to JSON automatically
    }
}

Added: the third way to control the status code (after ResponseEntity and the default 200) is the @ResponseStatus annotation — a very common interview follow-up: "can you return 201 without using ResponseEntity?"

On a handler method — return the POJO directly, the annotation supplies the status:

@PostMapping
@ResponseStatus(HttpStatus.CREATED)      // 201 instead of the default 200
public User createUser(@RequestBody User user) {
    return userService.save(user);       // body is still auto-serialized to JSON
}

On a custom exception class — whenever this exception escapes a controller, Spring answers with the declared status (no handler code needed):

@ResponseStatus(value = HttpStatus.NOT_FOUND, reason = "User not found")
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(long id) { super("User " + id + " not found"); }
}

Trade-offs worth saying out loud in an interview: @ResponseStatus is static — one fixed status, and per-request headers/bodies are impossible. Use ResponseEntity when the status depends on runtime logic (found → 200, missing → 404), and @ResponseStatus when it never changes. If both are present, the ResponseEntity's status wins.

Response Codes — the five families

The notes draw the codes as a number line from 1XX to 5XX:

  • 1XX — Informational — 100
  • 2XX — Success — 200, 201, 202, 204, 206
  • 3XX — Redirection — 301, 308, 304
  • 4XX — Validation error (client side) — 400, 401, 403, 404, 405, 409, 422, 429
  • 5XX — Server error — 500, 501, 502

2XX (Success)

Request received from the client is received and processed successfully.

  • 200 — OK — GET, POST — Request is successful and we are returning the response body.
  • 201 — Created — POST — Request is successful and a new resource is created.
  • 202 — Accepted — POST — Request is successfully accepted but processing is not yet completed. Batch processing like Export / Import etc.
  • 204 — No Content — DELETE — Request is successful and we are NOT returning any data in the response body.
  • 206 — Partial Content — POST — Request is partially successful — say Bulk Addition of 100 users, 95 passed and 5 requests failed; this response code can be used.

3XX (Redirection)

Client must take additional action to complete the request.

  • 301 — Moved Permanently — When we migrate from a legacy API to a new API (old status code — the new one is 308) — All requests should be directed to the new URI: .header("Location", "/v1.0/abc"). Clients like Postman have a setting "Automatically follow redirects" → ON/OFF.
  • 308 — Permanent Redirect — When we migrate from a legacy API to a new API — Same as 301, but it does not allow the HTTP method to change while redirecting (e.g. if the old API call is POST, then the new API call should also be POST — which is relaxed in 301).
@GetMapping("/old-api")
public ResponseEntity<Void> oldApi() {
    return ResponseEntity.status(HttpStatus.PERMANENT_REDIRECT)   // 308
            .header("Location", "/v1.0/abc")
            .build();
}

304 — Not Modified (GET)

Avoid using it with PATCH. Example: you are trying to update the name of the user, but say the name is already the same in the DB, so no update is required. In that case we should not throw 304, instead return 204 or 200.

The caching flow with If-Modified-Since:

  1. Client makes a GET call; the server returns it with the Last-Modified time in the header.
  2. Client caches the response.
  3. Client makes the GET call again, passing this last-modified time in the If-Modified-Since header.
  4. Server checks the particular resource's last update time against what the client provided — if the resource is not updated, the server simply returns 304 (NOT_MODIFIED).
  5. If modified, the server processes the request as usual and returns the new values.
// Spring does the If-Modified-Since comparison for you:
@GetMapping("/user/{id}")
public ResponseEntity<User> getUser(@PathVariable long id, WebRequest request) {
    long lastModified = userService.lastModified(id);
    if (request.checkNotModified(lastModified)) {
        return null;                     // Spring already wrote 304 Not Modified
    }
    return ResponseEntity.ok().lastModified(lastModified).body(userService.find(id));
}

Added: the modern sibling of this flow uses an ETag (a version/hash of the resource) instead of a timestamp — no clock precision problems, works for sub-second updates. Server sends ETag: "v5"; client sends it back as If-None-Match: "v5"; same checkNotModified API:

@GetMapping("/user/{id}")
public ResponseEntity<User> getUser(@PathVariable long id, WebRequest request) {
    String etag = "\"v" + userService.version(id) + "\"";   // e.g. the @Version column
    if (request.checkNotModified(etag)) {
        return null;                     // Spring wrote 304 + the ETag header
    }
    return ResponseEntity.ok().eTag(etag).body(userService.find(id));
}

For hash-based ETags across all GETs you can simply register Spring's ShallowEtagHeaderFilter — it computes an MD5 of the response body and answers 304 when it matches (saves bandwidth, not server CPU).

4XX (Client / Validation Error)

  • 400 — Bad Request — GET, POST, PATCH, DELETE — Client is not passing the request details needed to process the request.
  • 401 — Unauthorised — GET, POST, PATCH, DELETE — Any API which requires authentication, and the client tries to access it without providing authentication details.
  • 403 — Forbidden — GET, POST, PATCH, DELETE — Say only ADMIN can perform a certain operation. If the API gets invoked by anyone apart from ADMIN we should throw the 401 [sic] status code, as the client does not have permission to access the resource.
  • 404 — Not Found — GET, PATCH, DELETE — The requested resource which the client passes is not found in the DB by the server. E.g. get the user detail for ID 123, but no such ID is present in the DB.
  • 405 — Method Not Allowed — GET, POST, PATCH, DELETE — E.g. hitting a GET API but with the POST HTTP method. In Spring Boot the DispatcherServlet might throw this error, as control does not even reach the Controller.
  • 409 — Conflict — POST, PATCH, DELETE — Creating the same resource twice: POST v1.0/user and again POST v1.0/user.
  • 422 — Un-processable Entity — GET, POST, PATCH, DELETE — Your application's business validation — e.g. a France user should not be allowed to open an account (as the country is not supported yet).
  • 429 — Too Many Requests — GET, POST, PATCH, DELETE — Say our rule is: 1 user can make 10 calls in a minute. If user 12345 makes the 11th call in a minute, that 11th call should fail and we can throw the 429 error code.

Added: the 403 row keeps the notes' original "throw 401" wording marked [sic] — it should of course read 403. The rule of thumb: 401 = who are you? (not authenticated), 403 = I know who you are, but you may not do this (not authorized).

5XX (Server Error)

Request failed at the server, even though the client passed a valid request. Means something is wrong at the server.

  • 500 — Internal Server Error — GET, POST, PATCH, DELETE — Generic code when no more specific error code is suitable.
  • 501 — Not Implemented — GET, POST, PATCH, DELETE — API lacks the ability to fulfill the request — or say the API is in development and will be available in the future.
  • 502 — Bad Gateway — GET, POST, PATCH, DELETE — Server acting as a proxy got an invalid response while calling upstream. Example: my app is deployed behind a reverse proxy (Nginx). If Nginx is not able to communicate with my app, it is eligible to throw 502 Bad Gateway.

Added: one more 5XX worth knowing — 503 Service Unavailable: the server is temporarily unable to handle the request (overloaded, or down for maintenance), often sent with a Retry-After header. Unlike 502 it is about this server being unavailable, not a bad upstream answer.

100 (Informational)

An interim response to communicate request progress or its status before processing the final request.

100 — Continue — POST. Before sending the (large) request, the client checks with the server if it can handle the request and is ready:

  1. Client adds a few things in the headers first: - Content-Length: 1048576 - Content-Type: multipart/form-data - Expect: 100-continue
  2. Server checks that Expect: 100-continue is present in the header — meaning the client is just checking. So the server validates everything (authentication, authorization, content type, length etc.).
  3. If the server is okay, it returns the 100 Continue status code.
  4. Client receives it and then invokes the API again without Expect, and the server processes the request.

Good to know: instead of building an error ResponseEntity by hand, Spring lets you throw ResponseStatusException from anywhere: throw new ResponseStatusException(HttpStatus.NOT_FOUND, "User 123 not found"); — Spring converts it into the proper status code + message. How Spring resolves exceptions into responses is exactly the next topic.

Architect's Corner

  • Standard error body: Spring 6 ships ProblemDetail (RFC 7807 / RFC 9457) — {type, title, status, detail, instance}. Adopt it org-wide instead of inventing error JSON shapes.
  • Idempotency by method: GET/PUT/DELETE must be idempotent; POST is not — support an Idempotency-Key header for safely-retryable POSTs (payments!).
  • 429 must carry Retry-After, and clients should honor it with exponential backoff + jitter.
  • 304 + ETag is cheap server-side caching for read-heavy APIs — If-None-Match beats If-Modified-Since (no clock issues).
  • Add 503 Service Unavailable to your vocabulary: the correct answer during graceful shutdown and dependency outages — load balancers react to it.

Quick Revision

  • Response = Status Code + Header + Body; ResponseEntity<T> builds it — Builder pattern: status(...)/headers(...) return the builder, body(...) (last!) returns the ResponseEntity; build() for no body (ResponseEntity<Void>); default status is 200.
  • @ResponseBody is required to return a plain string/POJO; @RestController adds it to all methods; plain @Controller treats the return value as a view.
  • Families: 1XX informational, 2XX success, 3XX redirection, 4XX validation/client error, 5XX server error.
  • 2XX: 200 OK, 201 Created, 202 Accepted (batch), 204 No Content (DELETE), 206 Partial Content (bulk 95/100).
  • 3XX: 301 Moved Permanently (method may change), 308 Permanent Redirect (method preserved), 304 Not Modified (Last-Modified → If-Modified-Since; don't misuse with PATCH — return 200/204).
  • 4XX: 400 bad request, 401 unauthenticated, 403 forbidden (authenticated but no permission), 404 not found, 405 method not allowed (DispatcherServlet), 409 conflict (duplicate POST), 422 business validation, 429 rate limit.
  • 5XX: 500 generic, 501 not implemented, 502 bad gateway (proxy ↔ upstream); plus 503 service unavailable.
  • 100 Continue: headers with Expect: 100-continue → server validates → 100 → client sends the real request.
  • Static shortcuts: ok / created(uri) / accepted / noContent / badRequest / notFound / unprocessableEntity; POST → 201 uses ResponseEntity.created(location) with ServletUriComponentsBuilder.
  • @ResponseStatus sets a fixed status on a handler method or a custom exception class — static; use ResponseEntity when the status depends on runtime logic.
  • ETag / If-None-Match = version-based 304 (beats timestamps); request.checkNotModified(etag) + .eTag(...), or ShallowEtagHeaderFilter app-wide.
