Memory Management — The Big Picture

When a Java program runs, the JVM (Java Virtual Machine) asks the computer's RAM for memory. The JVM splits this memory into two main areas:

  1. Stack memory
  2. Heap memory

Both are created by the JVM and stored in RAM.

Simple analogy: think of the stack as your small work desk — you keep only the papers you are working on right now, and you clear them as soon as you finish. The heap is the big storeroom — everything big and long-living goes there, and a cleaner (the Garbage Collector) throws away things nobody uses anymore.

Stack Memory

The stack stores the small, short-lived things:

  • Temporary (local) variables — variables declared inside a method.
  • A separate memory block for each method call. Every time a method is called, a new "frame" (block) is pushed on top of the stack. When the method finishes, that frame is removed.
  • Primitive data types (int, char, boolean, double, …) declared inside methods live directly on the stack.
  • References to heap objects. The object itself lives in the heap, but the "address label" (reference) pointing to it lives on the stack.

More important rules about the stack:

  • Each thread has its own stack memory. Threads never share stacks — this is why local variables are automatically thread-safe.
  • Scope rule: a variable is visible only inside its own scope (its block/method). As soon as the variable goes out of scope, it is deleted from the stack automatically. No garbage collector is needed for the stack.
  • When stack memory becomes full, Java throws java.lang.StackOverflowError.
public class MemoryDemo {
    public static void main(String[] args) {
        int a = 10;                         // primitive -> stored on STACK
        String s = new String("hello");     // reference 's' on STACK,
                                            // the String object on HEAP
        printMessage(s);                    // new stack frame is created
    }                                       // frames removed when methods end

    static void printMessage(String msg) { // 'msg' is a reference on STACK
        System.out.println(msg);
    }
}

Output:
hello

Example — StackOverflowError

If a method keeps calling itself and never stops, frames keep piling up until the desk (stack) overflows:

public class StackOverflowDemo {
    static void callMyself() {
        callMyself();   // no stopping condition -> infinite recursion
    }

    public static void main(String[] args) {
        callMyself();
    }
}

Output:
Exception in thread "main" java.lang.StackOverflowError

Remember: Stack = small, fast, per-thread, cleans itself automatically. Full stack → StackOverflowError.

Three Kinds of References (Strong, Weak, Soft)

The stack holds references to heap objects. Java has different "strengths" of references, which tell the Garbage Collector how important the object is:

  • Strong Reference — The normal reference: Person p = new Person();. "I am using this — do NOT touch it." — Never, as long as the strong reference exists
  • Weak Reference — "You may throw this away whenever you like." — As soon as the GC runs, it frees the space
  • Soft Reference — "Throw it away only if you really need the space." — Only when memory is running low
import java.lang.ref.SoftReference;
import java.lang.ref.WeakReference;

public class ReferenceDemo {
    public static void main(String[] args) {
        // 1. Strong reference — GC will never collect this while 'strong' exists
        String strong = new String("I am strong");

        // 2. Weak reference — collected as soon as GC runs
        WeakReference<String> weak =
                new WeakReference<>(new String("I am weak"));

        // 3. Soft reference — collected only when memory is low
        SoftReference<String> soft =
                new SoftReference<>(new String("I am soft"));

        System.out.println(weak.get());  // object still there (GC not run yet)
        System.gc();                     // politely ask GC to run
        System.out.println(weak.get());  // very likely null now
        System.out.println(soft.get());  // still alive (memory is not low)
    }
}

Output (typical):
I am weak
null
I am soft

Heap Memory

The heap stores the big, long-lived things:

  • All objects created with new live in the heap.
  • There is no fixed order for allocating heap memory — objects are placed wherever there is space (unlike the neat, ordered stack).
  • The Garbage Collector (GC) deletes objects from the heap that no one references anymore (unreferenced objects). It uses the Mark and Sweep algorithm (explained below).
  • Heap memory is shared by all threads (unlike the stack). That is why shared objects need synchronization.
  • The heap also contains the String Pool — the special area where string literals like "hello" are kept and reused.
  • When heap memory becomes full, Java throws java.lang.OutOfMemoryError.

Remember: Stack full → StackOverflowError. Heap full → OutOfMemoryError. This is a very common interview question!

Stack vs Heap — quick comparison

  • Stores — Local variables, primitives, references, method frames — Objects (and the String Pool)
  • Order — Ordered (LIFO — last in, first out) — No particular order
  • Shared? — No — each thread has its own stack — Yes — shared by all threads
  • Cleanup — Automatic — variable deleted when it goes out of scope — Garbage Collector removes unreferenced objects
  • Error when full — java.lang.StackOverflowError — java.lang.OutOfMemoryError
  • Size — Small — Large

Heap Generations — How the Heap Is Organized

The heap is further divided into areas called generations:

  • Young Generation — where new objects are born. Minor GC happens here.
  • Eden — brand-new objects start here.
  • Survivor spaces: S0 and S1 — objects that survive a GC round move here.
  • Old / Tenured Generation — long-living objects end up here. Major GC happens here.

Beside the heap there is a non-heap area:

  • Metaspace (before Java 8 this was called PermGen). It stores:
  • constants
  • class (static) variables
  • class metadata (information about the classes themselves)
 +---------------------------------------------+     +---------------+
 |                HEAP MEMORY                  |     |   METASPACE   |
 |  +--------------------+ +----------------+ |     |  (Non-Heap;   |
 |  |  YOUNG GENERATION  | | OLD / TENURED  | |     |  was PermGen  |
 |  |  +------+---+---+  | |  GENERATION    | |     |  before Java8)|
 |  |  | Eden | S0| S1|  | |                | |     |  - constants  |
 |  |  +------+---+---+  | |  (Major GC)    | |     |  - class vars |
 |  |     (Minor GC)     | |                | |     |  - class      |
 |  +--------------------+ +----------------+ |     |    metadata   |
 +---------------------------------------------+     +---------------+

In plain words: the heap is one long box split into Young Generation and Old Generation. The Young Generation itself is split into Eden and two Survivor spaces (S0 and S1). Metaspace sits outside the heap.

School analogy

Think of objects as students and GC runs as exams:

  • Eden = the first grade. Every new student (object) starts here.
  • S0 / S1 = higher grades. Each time a student passes an exam (survives a GC because someone still references it), it is promoted and its age goes up by 1.
  • Old Generation = college. After passing enough exams (reaching the age threshold), the student is promoted permanently.
  • Students who fail (objects with no reference) are removed from school (deleted from memory).

How Minor GC and promotion actually work — step by step

The notes give a concrete example with 8 objects (01 … 08). The Minor GC in the Young Generation runs very frequently (in the notes' example: every ~6 seconds), while the Major GC in the Old Generation runs very rarely, periodically.

 Objects created: 01, 02, 03, 04, 05, 06, 07, 08

 STEP 1: 01..05 are in Eden.  GC runs (Mark & Sweep).
         02 and 05 have NO reference  -> deleted.
         01, 03, 04 survive           -> moved to S0 with age 1.

 STEP 2: New objects 06, 07, 08 fill Eden. GC runs again.
         06, 07 have NO reference, and now 04 has NO reference -> deleted.
         Survivors move to the *other* survivor space (S1),
         and each survivor's age +1:  01(age 2), 08(age 1), ...

 STEP 3: GC runs again. 08 and 03 have NO reference -> deleted.
         01 survives again -> age 3.

 STEP 4: Age 3 = THRESHOLD  ->  PROMOTION!
         01 is moved from the Young Generation to the OLD GENERATION.

 +--------+------+------+     +------------------+
 |  Eden  |  S0  |  S1  |     |  Old Generation  |
 |        |      |      | --> |       01         |
 +--------+------+------+     +------------------+
        Minor GC                   Major GC

Key points to remember from this walk-through:

  1. GC is invoked and runs the Mark & Sweep algorithm.
  2. Objects with no reference are removed immediately (like 02 and 05 in step 1).
  3. Objects that survive are moved between the survivor spaces S0 ↔ S1, and their age increases by 1 each time they survive.
  4. When an object's age reaches the threshold (age 3 in the notes' example), it gets promoted to the Old Generation.
  5. Cleaning the Young Generation = Minor GC (fast, frequent). Cleaning the Old Generation = Major GC (slower, rare).

Example — OutOfMemoryError

import java.util.ArrayList;
import java.util.List;

public class OutOfMemoryDemo {
    public static void main(String[] args) {
        List<int[]> list = new ArrayList<>();
        while (true) {
            list.add(new int[1_000_000]);  // keep filling the heap forever
        }
    }
}

Output (after some time):
Exception in thread "main" java.lang.OutOfMemoryError: Java heap space

Mark and Sweep with Compact Memory

This is the core algorithm the Garbage Collector uses. It has three simple steps:

  1. Mark — the GC walks through memory and puts an "X" on every object that has no reference (nobody can reach it anymore). These are the objects to be removed.
  2. Sweep — all the marked objects are deleted. This frees their memory, but it leaves gaps (holes) scattered around.
  3. Compact — the remaining live objects are shifted together to the start of the memory, so all the free space becomes one big continuous block. This makes it fast to allocate new (even big) objects.
 Before:    [A][B][C][D][E][F]        (B, D, E have no references)

 1) MARK:   [A][X][C][X][X][F]        mark which to be removed

 2) SWEEP:  [A][ ][C][ ][ ][F]        remove them -> gaps appear

 3) COMPACT:[A][C][F][ ][ ][ ]        shift survivors together
                                      -> free space is now continuous

Analogy: imagine a bookshelf. First you put stickers on the books nobody reads (mark), then you throw those books away (sweep), and finally you slide the remaining books to one side so all the empty space is together (compact).

Types of Garbage Collectors

There are four classic types of Garbage Collectors:

  1. Serial GC
  2. Parallel GC — the default (in classic JVM versions)
  3. CMS (Concurrent Mark Sweep)
  4. G1 (Garbage First)

An important term first: "Stop the World" pause — when the GC starts, all application threads are paused so the GC can clean safely. A long pause makes your app freeze, so newer GCs try to shorten or remove it.

1) Serial GC

  • Only 1 thread does the garbage collection work (for both Minor and Major GC).
  • When GC starts, all application threads are paused (full stop-the-world).
  • Simple, but slow — like a school with one cleaner who locks all the classrooms while cleaning.

2) Parallel GC (the default)

  • Same idea as Serial, but multiple GC threads clean at the same time.
  • Application threads are still paused, but the pause is shorter because many cleaners work together.

3) CMS — Concurrent Mark Sweep

  • While the application threads are working, the GC thread also works at the same time (concurrently).
  • But this is not 100% guaranteed — sometimes it still needs short pauses.
  • No memory compaction — after sweeping, the gaps stay, so memory can become fragmented.

4) G1 — Garbage First

  • Goal: application threads are (almost) 100% never stopped — pauses are tiny and predictable.
  • Plus it does compaction, so memory does not get fragmented.
  • The best of both worlds: concurrent like CMS and compacting like Serial/Parallel.

GC types — comparison table

  • Serial — 1 — Yes — fully paused — Yes — One cleaner, everything stops
  • Parallel (default) — Many — Yes — but shorter pause — Yes — Many cleaners, everything still stops
  • CMS — Runs alongside app — Mostly no (not 100% sure) — No — Cleans while app runs, but leaves gaps
  • G1 — Runs alongside app — ~100% no stopping — Yes — Cleans while app runs AND tidies memory

Remember: Serial = 1 thread + full pause. Parallel = many threads + pause (default). CMS = concurrent but no compaction. G1 = concurrent + compaction.

Quick Revision

  • JVM memory = Stack + Heap, both created by the JVM in RAM.
  • Stack: local variables, primitives, references, one frame per method call; each thread has its own stack; variables die when they go out of scope; full stack → StackOverflowError.
  • References come in three strengths: Strong (never collected while held), Weak (collected as soon as GC runs), Soft (collected only when memory is low).
  • Heap: all objects + the String Pool; shared by all threads; no allocation order; full heap → OutOfMemoryError.
  • Heap generations: Eden → S0/S1 (Minor GC, age +1 each survival) → Old/Tenured Generation (Major GC) once age hits the threshold. Metaspace (non-heap, was PermGen before Java 8) stores constants, class variables and class metadata.
  • GC algorithm = Mark (tag unreachable) → Sweep (delete) → Compact (shift survivors together).
  • Four GC types: Serial (1 thread, stop-the-world), Parallel (many threads, default), CMS (concurrent, no compaction), G1 (concurrent + compaction).
