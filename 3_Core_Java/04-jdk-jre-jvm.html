Java's Superpower: Platform Independence

First, four quick facts about Java:

  • Java is a platform independent language — the same program runs on Windows, Linux, or Mac without any change.
  • Java is an object-oriented programming language.
  • WORA — Write Once, Run Anywhere. You write your code one time, and it can run on any machine.
  • Java is a portable language — your compiled program can be carried to any computer and it just works.

Everyday analogy: think of bytecode as a message written in a universal language, and the JVM as a local translator. You write your letter once (WORA). Every country (operating system) has its own translator (JVM) who converts the universal letter into the local language (machine code). You never rewrite the letter — you just need the right translator on each machine.

To understand how this magic works, we need 3 important concepts:

  • JVM — Java Virtual Machine
  • JRE — Java Runtime Environment
  • JDK — Java Development Kit

How They Fit Together (Nested Circles)

The notes show three nested circles: JDK is the biggest circle, JRE sits inside it, and JVM sits innermost.

+-----------------------------------------------+
|                    JDK                        |
|   (compiler javac, debugger, language...)     |
|   +---------------------------------------+   |
|   |                 JRE                   |   |
|   |   (class libraries: java.lang, ...)   |   |
|   |      +------------------------+       |   |
|   |      |          JVM           |       |   |
|   |      |  (runs the bytecode)   |       |   |
|   |      +------------------------+       |   |
|   +---------------------------------------+   |
+-----------------------------------------------+

In plain words: JVM is inside JRE, and JRE is inside JDK. Every JDK contains a JRE, and every JRE contains a JVM.

  • JVM = the engine that actually runs your program.
  • JRE = JVM + the libraries needed to run programs.
  • JDK = JRE + the tools needed to write and compile programs.

JVM — Java Virtual Machine

  • JVM stands for Java Virtual Machine.
  • It is an abstract machine — it is not physically present. You cannot touch it; it is a program that behaves like a computer inside your computer.
  • The JVM's job: take bytecode and convert it into machine code that your actual CPU can execute.
 Java Program --> Compiler --> ByteCode --> JVM --> Machine Code --> CPU --> Output
|__________________________________________|
        Platform Independent part

Step by step, in plain words:

  1. Java Program — you write human-readable code in a .java file.
  2. Compiler (javac) — translates your code into bytecode.
  3. ByteCode — a universal, intermediate form of your program. Everything up to here is platform independent — the same bytecode works everywhere.
  4. JVM — reads the bytecode and converts it to machine code for the specific machine it is running on.
  5. Machine Code — raw instructions your processor understands.
  6. CPU — executes the machine code.
  7. Output — you see the result on your screen.

Two very important points:

  • Bytecode is platform independent, but the JVM itself is platform dependent — there is a separate JVM built for Windows, another for Linux, another for Mac. That is exactly how "write once, run anywhere" works: one bytecode, many JVMs.
  • The JVM contains a JIT (Just-In-Time) Compiler — a smart part of the JVM that compiles frequently used bytecode into machine code on the fly, making your program run faster.

Compiling a real file

If your file is student.java, this is what happens when you compile and run it:

student.java --(run: javac student.java)--> ByteCode (student.class)

// student.java
public class student {
    public static void main(String[] args) {
        System.out.println("Hello from bytecode!");
    }
}

Commands and result:

$ javac student.java     // compiler creates the bytecode file: student.class
$ java student           // JVM runs the bytecode
Hello from bytecode!

Remember: javac turns .java into .class (bytecode). The JVM turns bytecode into machine code. Bytecode = platform independent; JVM = platform dependent.

JRE — Java Runtime Environment

  • JRE stands for Java Runtime Environment.
  • The JRE contains the JVM plus the class libraries (ready-made code such as java.lang, java.util, etc.). These libraries give you String, System.out.println, ArrayList and thousands of other tools for free.
  • If you have bytecode, you can run it using the JRE — but you cannot write (develop) code with the JRE alone. It has no compiler.

Analogy: the JRE is like a music player. It can play songs (run bytecode), but it cannot record new songs (write and compile code).

JDK — Java Development Kit

  • JDK stands for Java Development Kit. It is the full toolbox for programmers. It contains:
  • The programming language itself (everything needed for your .java files)
  • The compiler (javac)
  • A debugger (tool to find and fix bugs step by step)
  • Plus the entire JRE inside it

The simple formula from the notes:

JDK = JRE + (Programming Language + Compiler + Debugger)
  • Only run a Java program (bytecode) — JRE (which includes JVM)
  • Write, compile, debug and run Java programs — JDK (which includes JRE)

Remember: Developers install the JDK. Users who only run Java programs need just the JRE.

Java Editions — JSE, JEE, JME

Java comes in three editions, each for a different kind of work:

  • JSE — Java Standard Edition — Core Java — the fundamentals (classes, OOP, collections, etc.)
  • JEE — Java Enterprise Edition — Large business/web applications
  • JME — Java Micro/Mobile Edition — Mobile and small-device applications
  • JSE is Core Java — what beginners learn first.
  • JEE = JSE + extra APIs. It adds enterprise features on top of standard Java, such as:
  • Transaction API (managing units of work safely)
  • Rollback (undoing a failed transaction)
  • Servlet (handling web requests)
  • JSP (JavaServer Pages, for building web pages)
  • JME provides APIs for mobile applications (small devices).

Why Can One Java File Have Only ONE Public Class?

Note: One Java file must have only one public class.

Also, a class can contain these members:

  • Variables
  • Methods
  • Constructor
  • Inner Class

The rules first

// File name: A.java
public class A {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}

Output:

Hello

Two rules work together here:

  1. The main method should be inside a public class.
  2. The public class name should be the same as the file name (public class A must live in A.java).

The reasons (in plain words)

Remember the flow: Java Program --Compile--> ByteCode --> the JVM calls the main() method.

Reason 1: The JVM calls the main method using the class name. So that class must be accessible from anywhere — which means it should be public. If the JVM could not reach the class, it could not start your program at all.

Reason 2: Suppose one file could hold two public classes:

// File name: Employee.java  — NOT allowed, will NOT compile!
public class A {
}
public class B {
}

Now the JVM has a puzzle: the file is called Employee.java, but which public class should it match with the file — A or B? Where should it look for main()? By forcing one public class per file, named the same as the file, the JVM does not have to worry about other public classes — it can find the right class instantly just from the file name.

Analogy: a file is like a house and the public class is its name plate at the front door. One house, one name plate, and the plate must match the address (file name). If a house had two different name plates, the postman (JVM) would never know whom to deliver to.

Remember: The main method lives inside the public class, and the public class name must equal the file name. That is how the JVM finds and starts your program.

Quick Revision

  • Java is platform independent, object-oriented, and portable — WORA: Write Once, Run Anywhere.
  • Nesting: JDK ⊃ JRE ⊃ JVM (JVM inside JRE, JRE inside JDK).
  • JVM = abstract machine (not physically present); converts bytecode → machine code; contains the JIT compiler; the JVM itself is platform dependent, while bytecode is platform independent.
  • Flow: Java Program → Compiler (javac) → ByteCode (.class) → JVM → Machine Code → CPU → Output.
  • JRE = JVM + class libraries (java.lang, java.util, ...); can run bytecode but cannot compile code.
  • JDK = JRE + programming language + compiler (javac) + debugger — the full developer toolbox.
  • Editions: JSE (Core Java), JEE = JSE + APIs (Transaction API, Rollback, Servlet, JSP), JME (mobile APIs).
  • A class can contain variables, methods, a constructor, and inner classes.
  • One file = one public class, named exactly like the file, holding main — so the JVM can find and call main() without confusion.
